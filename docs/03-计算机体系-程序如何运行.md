## 1. 📌 冯诺依曼结构——计算机的骨架

### 🏢 生活类比：厨房

```
CPU（中央处理器）= 厨师
  → 拿菜谱（指令）→ 取食材（数据）→ 做菜（计算）→ 上菜（结果）

内存（Memory）= 操作台
  → 食材和菜谱都放在操作台上
  → 离厨师近，存取快，但关了炉子（断电）就没了

硬盘（Disk）= 冰箱
  → 食材的大仓库，断电也在，但存取慢

总线（Bus）= 厨师的手和脚
  → 从操作台取东西（读内存）、放东西到操作台（写内存）
```

### 冯诺依曼结构的核心思想

```
┌───────────────────────────────────────────────────┐
│                    冯诺依曼结构                       │
│                                                    │
│   ┌──────────┐      总线(Bus)      ┌──────────┐     │
│   │          │←──────────────────→│          │     │
│   │   CPU    │   取指令/取数据/存结果  │  内存    │     │
│   │          │                    │          │     │
│   │ 寄存器组  │                    │ 指令和数据 │     │
│   │ ALU运算器│                    │ 混在一起放 │     │
│   │ 控制单元 │                    │          │     │
│   └──────────┘                    └──────────┘     │
│                                                    │
│   程序 = 指令 + 数据，都放在内存里                       │
│   CPU 从内存读一条指令 → 执行 → 读下一条 → 循环          │
└───────────────────────────────────────────────────┘
```

**最重要的结论：** 指令和数据在内存里没有区别——都是一堆字节。CPU 把它们当指令执行就是指令，当数据读就是数据。这也就是为什么**黑客能把数据当成指令来执行**（shellcode 的原理）。

## 2. 📌 寄存器——CPU的临时草稿纸

### 🏢 生活类比

```
寄存器 = 厨师手边的调料小碗
  → 不是从大仓库拿，而是放在手边，随拿随用
  → 数量很少（x86-64有16个通用寄存器），大小固定（64位 = 8字节）
```

### x86-64 核心寄存器一览

```
┌─────────────────────────────────────────────────────────┐
│                    x86-64 寄存器地图                      │
│                                                          │
│  RAX  ┌──────────────────────────64位────────────────┐   │
│       │<─── EAX (低32位) ────>                       │   │
│       │<── AX (低16位) ──>│                          │   │
│       │<─AH─><──AL──>│  │  │                          │   │
│       └──────────────────────────────────────────────┘   │
│                                                          │
│  通用寄存器（做什么用由程序定）:                              │
│  RAX  — 函数返回值（最重要！call一个函数后，返回值就在RAX里）   │
│  RBX  — 被调用者保存的通用寄存器                             │
│  RCX  — 函数调用的第4个参数                                 │
│  RDX  — 函数调用的第3个参数                                 │
│  RSI  — 函数调用的第2个参数（Source Index）                 │
│  RDI  — 函数调用的第1个参数（Destination Index）            │
│  R8   — 第5个参数                                         │
│  R9   — 第6个参数                                         │
│                                                          │
│  特殊寄存器（CPU 自己用）:                                   │
│  RSP  — ☆ 栈顶指针（Stack Pointer），永远指向栈的顶部        │
│  RBP  — ☆ 栈底指针（Base Pointer），当前栈帧的基准位置        │
│  RIP  — ★ 指令指针（Instruction Pointer），存放下一条指令地址  │
│          （这就是"返回地址"存的东西——改RIP就是改程序的流向！）   │
│  RFLAGS — 标志位（比较结果：谁大谁小、是否为零）               │
└─────────────────────────────────────────────────────────┘
```

### 🔥 网安人必须刻在脑子里的

```
RIP — 程序下一步去哪（把它改成哪，程序就去哪——ROP攻击的核心）
RSP — 栈的当前位置（push/pop 会改变它）
RBP — 当前函数的"栈底"参照物（局部变量都是 "rbp - 若干字节"）
RAX — 函数返回值放在这
RDI, RSI, RDX, RCX, R8, R9 — 传函数的参数（按顺序放）
```

## 3. 指令执行周期——CPU的呼吸

CPU 执行程序就是三个步骤的无限循环：

```
   ┌──────────────────────────────┐
   │        指令周期（约每纳秒一次）   │
   │                               │
   │   ① 取指(Fetch)                │
   │     ├→ 看 RIP 指向的内存地址      │
   │     ├→ 从那里取出下一条指令       │
   │     └→ RIP ← RIP + 指令长度     │
   │                               │
   │   ② 译码(Decode)               │
   │     └→ 理解这条指令是什么意思      │
   │                               │
   │   ③ 执行(Execute)              │
   │     └→ 真的去做（运算、读内存、写内存、跳转）│
   │                               │
   └───────────→ 回到① ←────────────┘
```

### 💻 看一个具体例子

```asm
; C代码: int x = a + b;

mov    eax, [rbp-4]    ; ① 把 a 从内存加载到 eax 寄存器（取数）
add    eax, [rbp-8]    ; ② 把 b 加到 eax 上（计算）
mov    [rbp-12], eax   ; ③ 把结果存回内存给 x（存结果）
```

## 4. 📌 栈帧结构——图解函数调用时到底发生了什么

**这是二进制安全最重要的一张图，请反复看。**

### 🏢 生活类比

```
函数调用 = 你正在看书（main），接到电话要处理一件事（func）

1. 你在当前页夹个书签（保存返回地址）—— "看完这一页处理完就回来"
2. 记下当前进度（保存旧的 rbp）
3. 开始处理电话里的事（执行 func）
4. 处理完，看看书签说看到哪了（取返回地址）
5. 把书翻回去继续看（跳回 main）
```

### 💻 函数调用的汇编过程

```c
// C 源代码
int add(int a, int b) {
    int result = a + b;
    return result;
}

int main() {
    int x = 3;
    int y = 4;
    int z = add(x, y);    // ← 调用发生在这里
    return 0;
}
```

```asm
; ===== main 中调用 add 的汇编 =====

; 1. 传参数（64位系统：前6个参数用寄存器）
mov    edi, [rbp-4]    ; 把 x(3) 放到 edi（第1个参数）
mov    esi, [rbp-8]    ; 把 y(4) 放到 esi（第2个参数）

; 2. CALL 指令（同时做两件事）
call   add              ; 等价于: ① push rip（存返回地址）
                        ;         ② jmp add（跳转到add函数）

; 3. add 返回后，返回值已在 eax 中
mov    [rbp-12], eax    ; 把返回值(7)存给 z

; ===== add 函数内部 =====
add:
    push   rbp          ; ④ 保存调用者(main)的 rbp
    mov    rbp, rsp     ; ⑤ 建立自己的栈帧——现在 rbp 指向新栈帧底部
    sub    rsp, 16      ; ⑥ 在栈上给局部变量留 16 字节空间

    mov    [rbp-4], edi ; ⑦ 把参数 a 存到栈上
    mov    [rbp-8], esi ; ⑧ 把参数 b 存到栈上

    mov    eax, [rbp-4] ; ⑨ eax = a
    add    eax, [rbp-8] ; ⑩ eax = a + b
    mov    [rbp-12], eax; ⑪ result = a + b

    mov    eax, [rbp-12]; ⑫ 返回值放 eax

    leave               ; ⑬ 等价于: mov rsp, rbp; pop rbp（恢复调用者栈帧）
    ret                 ; ⑭ 等价于: pop rip（取出返回地址，跳回去）
```

### 图解调用 add(3, 4) 时的栈变化

```
调用 add 之前(main的栈)：        进入 add 之后：
                                
高地址                          高地址
┌──────────────┐                ┌──────────────┐
│  main 的 y   │← rbp-8         │  main 的 y   │
├──────────────┤                ├──────────────┤
│  main 的 x   │← rbp-4         │  main 的 x   │
├──────────────┤                ├──────────────┤
│  旧的 rbp     │                │  旧的 rbp     │
├──────────────┤                ├──────────────┤
│ ★返回地址     │← call 压入      │  返回值       │← rip存放的位置
├──────────────┤                ├──────────────┤
│  ...         │← rsp           │  main的旧rbp  │← rbp(当前栈底)
                               ├──────────────┤
                               │  result=7    │← rbp-12
                               ├──────────────┤
                               │  b=4         │← rbp-8  (来自 esi)
                               ├──────────────┤
                               │  a=3         │← rbp-4  (来自 edi)
                               ├──────────────┤
                               │  ...         │← rsp
                               低地址
```

### 🔥 为什么能栈溢出？再看这张图

```
假设在 add 里有个 char buf[8] 而不是普通变量：

高地址
┌──────────────┐
│  旧的 rbp     │            ← 当 buf 写超过 8 字节时：
├──────────────┤               先覆盖 buf 后面的局部变量
│★ 返回地址     │            ← 再覆盖这个！（覆盖它 = 控制程序流向）
├──────────────┤
│  旧 rbp       │            ← 再覆盖这个
├──────────────┤
│  buf[7]      │ ← 这是 buf 的最后一个字节
│  buf[6]      │
│  ...         │ ← buf 从低地址向高地址增长
│  buf[0]      │ ← 这是 buf 的第一个字节
└──────────────┘
低地址
```

**buf 往高地址溢出的过程中，必然会经过"返回地址"。** 这就是栈溢出的全部秘密。

## 5. 📌 调用约定——函数之间怎么传话

不同的平台、不同的编译器，传参数的方式不同。

### x86-64 的 System V 调用约定（Linux 标准）

```
传参顺序（前6个用寄存器，多余的用栈）:

  第1个参数 → RDI
  第2个参数 → RSI
  第3个参数 → RDX
  第4个参数 → RCX
  第5个参数 → R8
  第6个参数 → R9
  第7个及以后 → 依次压入栈（从右往左）

  返回值 → RAX

  被调用者必须恢复的寄存器: RBX, RBP, R12-R15
  调用者自己保存的寄存器: RAX, RCX, RDX, RSI, RDI, R8-R11
```

### 32位 x86 的 cdecl 调用约定（对比）

```
传参方式：全部用栈！从右往左压入。
清理栈：调用者负责。

push 4        ; 第2个参数
push 3        ; 第1个参数
call add
add esp, 8    ; 调用者清理栈
```

**网安意义：** 32位和64位的传参方式完全不同，攻击手法也不同（32位用栈传参更好控制）。

## 6. 汇编速查——x86-64核心指令

### 数据移动

| 指令 | 含义 | 例子 |
|------|------|------|
| `mov a, b` | 把 b 的值复制给 a | `mov eax, 10` — eax=10 |
| `push x` | 把 x 压入栈顶 | `push rbp` — RSP-8, 把rbp存到新栈顶 |
| `pop x` | 从栈顶取值到 x | `pop rbp` — 从栈顶取值赋给rbp, RSP+8 |
| `lea dst, [addr]` | 取地址（不访问内存） | `lea rax, [rbp-4]` — rax = rbp-4 的地址 |

### 算术运算

| 指令 | 含义 |
|------|------|
| `add a, b` | a = a + b |
| `sub a, b` | a = a - b |
| `inc a` | a = a + 1 |
| `dec a` | a = a - 1 |

### 跳转（控制流）

| 指令 | 含义 |
|------|------|
| `jmp addr` | 无条件跳转（goto） |
| `call func` | 调用函数（push 返回地址; jmp func） |
| `ret` | 返回（pop rip——从栈顶取地址跳过去） |
| `cmp a, b` | 比较 a 和 b（设置标志位） |
| `je addr` | 如果相等就跳（jump if equal） |
| `jne addr` | 如果不相等就跳 |
| `jg addr` | 如果大于就跳（有符号） |
| `jl addr` | 如果小于就跳（有符号） |

### 🔥 最值得记住的三条

```asm
call X    =  push rip  +  jmp X        ; 调用函数
ret       =  pop rip                   ; 函数返回——从栈上取地址跳转
leave     =  mov rsp, rbp  +  pop rbp  ; 恢复调用者栈帧
```

**如果你能理解这三条指令的等价实现，栈溢出的攻击原理就全通了。**

## 7. 操作系统管的三件事

### 🏢 生活类比

```
操作系统 = 写字楼物业管理员

  1. 进程管理 = 安排谁用哪个办公室、用多久
      → 电脑同时开浏览器、编辑器、终端，每个都是一个进程
      → OS 负责让它们轮流用 CPU（时间片轮转）

  2. 内存管理 = 分配给每个公司（进程）专属办公区
      → 每个进程以为自己独占内存（虚拟内存的假象）
      → 实际物理内存是共享的，OS 负责映射

  3. 文件系统 = 管理档案柜
      → 文件存在哪、怎么找（inode、路径）
      → 谁能看、谁能改（权限：rwx、owner/group/other）
```

### 虚拟内存——每个进程的幻觉

```
每个进程以为自己在用全部内存：

进程A的内存视图:            实际物理内存:
┌──────────────┐           ┌──────────────┐
│ 0x0000~0xFFFF│ ← 以为   │ 真实物理页    │
│   (独占!)    │   独占    │ A页 B页 A页   │ ← OS 混着放
└──────────────┘           │ B页 A页 B页   │
                           └──────────────┘

内存虚拟化 = 一面"哈哈镜"，让每个进程觉得内存是自己的。
```

**网安意义：** 很多攻击涉及打破这种隔离——比如一个进程读另一个进程的内存。
