## 1. 📌 冯诺依曼结构——计算机的骨架

### 🏪 生活场景：餐厅后厨的运作方式

想象你走进一家餐厅的后厨：

```
厨师（CPU）站在操作台前。
操作台（内存）上摊着一本菜谱（程序）和一堆食材（数据）。
冰箱（硬盘）里存着更多食材，但取一次要花时间。
厨师的手（总线）在操作台和冰箱之间搬运东西。

厨师的工作循环：
  1. 看一眼菜谱当前这一行："打两个鸡蛋"
  2. 理解这是什么意思
  3. 从操作台上拿鸡蛋，打散，倒进锅里
  4. 看菜谱下一行："加盐少许"
  → 回到第 1 步
```

这个场景就是冯诺依曼结构的全部。没有魔法，没有"人工智能"——**计算机就是一个不断"取指令→理解→执行→取下一条"的机器，每秒做几十亿次。**

### 🔍 技术原理

冯诺伊曼结构由五个核心部件组成：

```
┌──────────────────────────────────────────────────────────┐
│                     冯·诺依曼结构                          │
│                                                           │
│   ┌──────────┐                    ┌──────────┐             │
│   │          │   地址总线(Address) │          │             │
│   │   CPU    │ ←────────────────→ │  内存    │             │
│   │          │   数据总线(Data)    │          │             │
│   │ ┌──────┐ │                    │ 指令和数据│             │
│   │ │寄存器│ │                    │ 混在一起放│             │
│   │ └──────┘ │                    │          │             │
│   │ ┌──────┐ │                    └──────────┘             │
│   │ │ ALU  │ │    运算器——做加减乘除、比较、逻辑运算           │
│   │ └──────┘ │                                              │
│   │ ┌──────┐ │                                              │
│   │ │控制单元│ │    取指令、译码、控制数据流动                  │
│   │ └──────┘ │                                              │
│   └──────────┘                                              │
│                                                           │
│   输入设备(键盘/鼠标/网卡) ──→ CPU ←── 输出设备(显示器/网卡)    │
│   硬盘(长久存储) ←─────────→ 内存 ←─────────→ CPU             │
└──────────────────────────────────────────────────────────┘
```

**最重要的结论：指令和数据在内存里没有区别——都是一堆字节。** CPU 把它们当指令执行就是指令，当数据读就是数据。内存地址 `0x401000` 处的一个字节 `0xC3`，如果你把它当数据读，它就是数字 195；如果 RIP（指令指针）指向它，CPU 就把它当 `ret` 指令来执行。

**这就是 shellcode 的原理：** 攻击者把自己构造的字节（数据）放到内存中，然后设法让 CPU 的 RIP 指向这些字节，CPU 就会把它们当成指令来乖乖执行。

### 💻 动手验证

```bash
# 任意一个程序都是字节组成的
xxd /bin/ls | head -5
# 输出: 00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............

# 这是 ELF（Linux可执行文件）的头部
# 前四个字节 7f 45 4c 46 = 0x7f 'E' 'L' 'F'
# 这既是数据（可以用眼睛读），也是程序（CPU 会执行后面的机器码）
```

## 2. 📌 寄存器——CPU 的临时草稿纸

### 🏪 生活场景：组装家具时的操作台

你在家组装一个宜家衣柜。说明书（程序）放在左边，螺丝木板（数据）散在面前。你的**手边只能同时放 5-6 个零件**。你需要不断在零件堆和手边空间之间搬运。

**寄存器就是手边那 5-6 个位置。** 数量极少（x86-64 有 16 个通用寄存器），但速度极快（1 个 CPU 周期就能读写）。内存相当于你身后的零件堆——空间很大（几个 GB），但每次转身去拿（读内存）要花几十到几百个 CPU 周期。

### 🔍 x86-64 核心寄存器全部解析

```
┌─────────────────────────────────────────────────────────────────┐
│                     x86-64 寄存器 完整地图                        │
│                                                                  │
│  64位:  RAX RBX RCX RDX RSI RDI RBP RSP R8-R15                  │
│         │   │   │   │   │   │   │   │   │                        │
│  32位:  EAX EBX ECX EDX ESI EDI EBP ESP R8D-R15D                │
│         │   │   │   │   │   │   │   │   │                        │
│  16位:  AX  BX  CX  DX  SI  DI  BP  SP  R8W-R15W               │
│         │   │   │   │   │   │   │   │   │                        │
│  8位:   AL  BL  CL  DL  SIL DIL BPL SPL R8B-R15B               │
│         AH  BH  CH  DH                                           │
│                                                                  │
│  ★ RAX  → 函数返回值放在这。call 一个函数回来，答案在 RAX 里       │
│  ★ RSP  → 栈顶指针。永远指向当前栈的"最顶上"（最低地址）           │
│  ★ RBP  → 栈底指针。当前函数的参照物，"局部变量 = RBP - N"         │
│  ★ RIP  → 指令指针。下一条要执行的指令的地址。                     │
│           ★ 这是攻击者最想控制的寄存器——控制 RIP 就控制了程序     │
│                                                                  │
│  传参（前6个参数）:                                               │
│  RDI = 第1个参数  ← "Destination Index"                          │
│  RSI = 第2个参数  ← "Source Index"                               │
│  RDX = 第3个参数                                                  │
│  RCX = 第4个参数                                                  │
│  R8  = 第5个参数                                                  │
│  R9  = 第6个参数                                                  │
│  第7个+ → 用栈传                                                  │
│                                                                  │
│  RFLAGS → 状态标志位：ZF(结果为零)、SF(结果为负)、CF(进位)          │
│           比较指令(cmp)就是在设置这些位，条件跳转(je/jg)在检测它们   │
└─────────────────────────────────────────────────────────────────┘
```

### 🔥 安全人必须刻在脑子里的四个寄存器

| 寄存器 | 全称 | 干嘛的 | 安全意义 |
|--------|------|--------|---------|
| RIP | Instruction Pointer | 下一条指令在哪 | ★ 控制了 RIP = 控制了程序。所有代码执行漏洞的终极目标 |
| RSP | Stack Pointer | 栈顶在哪 | push/pop/call/ret 都在操作 RSP |
| RBP | Base Pointer | 当前栈帧参照物 | 局部变量都是 `[rbp-N]`，溢出会覆盖旧的 RBP |
| RAX | Accumulator | 函数返回值 | ret2libc 攻击中 system() 的返回值放在这 |

### 💻 动手看寄存器

```bash
# 写一个最简单的程序
echo 'int main() { return 42; }' > ret42.c
gcc -g -o ret42 ret42.c

# 在 GDB 里看寄存器
gdb ./ret42 -q
(gdb) start              # 开始并停在 main
(gdb) i r                # 查看所有寄存器
(gdb) ni                 # 执行一条指令
(gdb) i r rax            # 看看 RAX 的值变化
(gdb) finish             # 执行到 main 返回
(gdb) i r rax            # RAX = 42 —— 这就是返回值！
```

## 3. 指令执行周期——CPU 的呼吸

### 🏪 生活场景：流水线工人

工厂流水线上有个工人，他的工作循环只有三步：

```
① 伸手从传送带上拿一个工件（取指 Fetch）
② 看看标签：这个要钻孔还是打磨（译码 Decode）
③ 操作机器完成工序（执行 Execute）
→ 回到①
```

**这个循环 CPU 每秒做 30-40 亿次（3-4 GHz）。** 每个步骤只需要不到 1 纳秒。你的 Python 程序最终也是被翻译成这些基本步骤来执行的。

### 🔍 一个具体例子

```asm
; C 代码: x = a + b;
; 假设 a 在 [rbp-4], b 在 [rbp-8], x 在 [rbp-12]

; CPU 做的事，分三步：
; ① mov eax, [rbp-4]     ; 取指: RIP→这条指令
;                           ; 译码: 这是 mov，从内存加载到寄存器
;                           ; 执行: 从地址 rbp-4 读 4 字节 → 放入 eax
;                           ; RIP = RIP + 3（这条指令长 3 字节）

; ② add eax, [rbp-8]     ; 取指: RIP→这条指令
;                           ; 译码: 这是 add，源操作数在内存
;                           ; 执行: eax = eax + [内存 rbp-8]
;                           ; RIP = RIP + 3

; ③ mov [rbp-12], eax    ; 取指: RIP→这条指令
;                           ; 译码: 这是 mov，目标在内存
;                           ; 执行: 把 eax 的值存入地址 rbp-12
;                           ; RIP = RIP + 3
```

每一条指令执行后，RIP 自动指向下一条——这就是程序"顺序执行"的本质。而 `call`、`jmp`、`ret` 这些指令就是**主动修改 RIP**，让 CPU 跳到别处执行。

## 4. 📌 栈帧结构——函数调用底层真相

> ⚠️ **这是连接 C 语言和汇编的桥梁。** 01 章你看到了栈溢出能覆盖返回地址。本章你来看"返回地址"到底是怎么被压入栈、怎么被取出来的——以及为什么 `call`/`ret` 就是对 RIP 和 RSP 的操作。

### 🏪 生活场景：你看书时接到电话

你在书房看一本小说（main 函数在执行），看到第 42 页。突然电话响了。

```
1. 你在第 42 页夹一个书签（保存返回地址到栈上）
   → call 指令做的事：push RIP（把下一条指令的地址压栈）

2. 你接起电话："喂？"（跳转到接电话的处理流程）
   → jmp func（RIP = func 的地址）

3. 你处理电话里的事（执行 func 的代码）
   → 新栈帧建立，局部变量分配

4. 处理完了，你挂电话（func 执行 ret）
   → pop RIP（从栈上取出之前保存的地址，放回 RIP）

5. 你翻到书签那一页（第 43 页），继续看书
   → RIP 回到了 call 的下一条指令
```

### 🔍 函数调用的完整汇编过程

```c
// 源代码
int add(int a, int b) {
    int result = a + b;
    return result;
}

int main() {
    int x = 3, y = 4;
    int z = add(x, y);   // ★ 调用发生在这里
    return 0;
}
```

对应的汇编（简化，64 位 Linux）：

```asm
; ===== main 函数中调用 add =====
section .text
main:
    ; ... main 的栈帧已建立 ...
    mov    DWORD PTR [rbp-4], 3     ; x = 3
    mov    DWORD PTR [rbp-8], 4     ; y = 4

    ; --- 准备调用 add(3, 4) ---
    mov    eax, DWORD PTR [rbp-8]   ; eax = y = 4
    mov    edx, eax                  ; edx = 4 (第3个参数？不——对)
    mov    eax, DWORD PTR [rbp-4]   ; eax = x = 3
    mov    esi, edx                  ; esi = 4 (第2个参数)
    mov    edi, eax                  ; edi = 3 (第1个参数)
    call   add                       ; ★ ① push RIP  ② jmp add

    ; --- add 返回后 ---
    mov    DWORD PTR [rbp-12], eax  ; z = add的返回值(在eax里)

; ===== add 函数内部 =====
add:
    push   rbp                       ; ① 保存 main 的 RBP
    mov    rbp, rsp                  ; ② 建立自己的栈帧：RBP = RSP
    sub    rsp, 16                   ; ③ 给局部变量分配 16 字节空间

    mov    DWORD PTR [rbp-20], edi  ; ④ 把第1个参数(3)存到栈上
    mov    DWORD PTR [rbp-24], esi  ; ⑤ 把第2个参数(4)存到栈上

    mov    edx, DWORD PTR [rbp-20]  ; edx = 3
    mov    eax, DWORD PTR [rbp-24]  ; eax = 4
    add    eax, edx                  ; eax = 3 + 4 = 7
    mov    DWORD PTR [rbp-4], eax   ; result = 7

    mov    eax, DWORD PTR [rbp-4]   ; 返回值 = 7 (放入 eax)

    leave                            ; ⑥ mov rsp, rbp; pop rbp
    ret                              ; ⑦ pop RIP (从栈上取出返回地址)
```

### 图解：调用 add(3, 4) 时栈的完整变化

```
调用前 (main 的栈)                   进入 add 后                      add 返回前

高地址                               高地址                          高地址
┌──────────────┐                     ┌──────────────┐                ┌──────────────┐
│ main 的 y=4  │ rbp-8               │ main 的 y=4  │                │ main 的 y=4  │
│ main 的 x=3  │ rbp-4               │ main 的 x=3  │                │ main 的 x=3  │
│ main 的旧RBP │                     │ main 的旧RBP │                │ main 的旧RBP │
├──────────────┤                     ├──────────────┤                ├──────────────┤
│ 返回地址     │ ← call 压入          │ 返回地址     │                │ 返回地址     │
│ (main 的     │                     │ (下一条指令)  │                │              │
│  下一条指令)  │                     ├──────────────┤                ├──────────────┤
│              │                     │ main 的旧RBP │ ← RBP 指向这   │ main 的旧RBP │
│ ...          │ ← RSP               ├──────────────┤                ├──────────────┤
│              │                     │ result=7     │ rbp-4          │ result=7     │
│              │                     │ b=4          │ rbp-24         │ b=4          │
│              │                     │ a=3          │ rbp-20         │ a=3          │
│              │                     │ ...          │ ← RSP          │ ...          │
低地址                               低地址                          低地址

                                                                    ret 后:
                                                                    RSP ↑ 弹走返回地址
                                                                    RIP = 返回地址
                                                                    RBP = main 的旧RBP
```

### 🔥 为什么栈溢出能控制程序——现在用汇编语言说

```
buf 在栈上的位置：RBP - 64（假设 64 字节）

攻击者往 buf 写了 100 字节：
  byte 0-63   → 填满 buf               ← 正常范围
  byte 64-71  → 覆盖 [rbp-?] 的变量     ← 破坏其他局部变量
  byte 72-79  → 覆盖 旧的 RBP            ← 破坏了栈帧链
  byte 80-87  → 覆盖 ★返回地址★          ← 控制了 ret 的目标！

当 func 执行 ret 指令时：
  ret = pop RIP = 从 RSP 指向的位置读 8 字节 → 写入 RIP
  这 8 字节是攻击者写入的！→ RIP 变成攻击者指定的值 → 程序被劫持！
```

## 5. 📌 调用约定——函数之间怎么传话

### 🏪 生活场景：公司之间的公文格式

中国公司和美国公司做生意。如果中国公司发中文合同，美国公司发英文合同——双方都得翻译，容易出错。于是国际商会制定了一个标准：国际贸易合同统一用英文，格式统一为"双方信息→条款→签名→日期"。

**调用约定就是函数之间的"标准合同格式"。** 它规定：
- 参数按什么顺序传、用寄存器还是用栈
- 返回值放哪
- 谁负责清理栈

### 🔍 x86-64 System V 调用约定（Linux 标准）

```
前 6 个整数/指针参数 → 依次放入: RDI, RSI, RDX, RCX, R8, R9
浮点参数 → 放入 XMM0-XMM7
第 7 个及以后的参数 → 从右向左压栈
返回值 → 放在 RAX（整数/指针）或 XMM0（浮点）

调用者负责: 保存 RAX, RCX, RDX, RSI, RDI, R8-R11（易失寄存器）
被调用者负责: 保存 RBX, RBP, R12-R15（调用结束后要恢复原值）
清理栈: 调用者负责（如果有栈传参的话）
```

### x86 (32位) cdecl 调用约定（对比）

```
所有参数 → 从右向左全部压栈！
返回值 → 放在 EAX
调用者负责清理栈（add esp, N）
```

**安全意义：32 位和 64 位的攻击手法完全不同。** 32 位全部用栈传参，你控制了栈就控制了参数；64 位前 6 个参数用寄存器，你需要先找到 `pop rdi; ret` 这样的 gadget 来设置寄存器。

### 💻 手写一个遵循调用约定的函数

```asm
; 在汇编里调用 printf("Hello %s\n", name)
; 遵循 System V 调用约定

section .data
    fmt db "Hello %s", 10, 0     ; 格式字符串
    name db "World", 0           ; 参数

section .text
    global main
    extern printf

main:
    ; 第1个参数: 格式字符串 → RDI
    lea rdi, [rel fmt]
    ; 第2个参数: name → RSI
    lea rsi, [rel name]
    ; AL = 0 (没有浮点参数)
    xor eax, eax
    call printf
    ; 返回值在 EAX 里
    ret
```

## 6. 汇编速查——x86-64 核心指令

### 💻 每条指令配一个生活类比

| 指令 | 含义 | 生活类比 |
|------|------|---------|
| `mov dst, src` | dst = src | 把东西从一个地方搬到另一个地方 |
| `push src` | RSP-=8; [RSP]=src | 把盘子放在盘子摞最顶上 |
| `pop dst` | dst=[RSP]; RSP+=8 | 从盘子摞最顶上取一个盘子 |
| `lea dst, [addr]` | dst = addr（取地址，不读内存） | 抄下门牌号，不进门 |
| `add a, b` | a = a + b | 往杯子里加水 |
| `sub a, b` | a = a - b | 从杯子里倒出水 |
| `cmp a, b` | 比较 a 和 b（设标志位） | 用秤称两个东西谁重 |
| `jmp addr` | RIP = addr | 跳到书里另一页 |
| `je addr` | 如果相等就跳 | 如果两袋米一样重，就去 A 仓库 |
| `jne addr` | 如果不相等就跳 | 重量不同就去 B 仓库 |
| `call func` | push RIP; jmp func | 在书上夹书签，翻到另一页 |
| `ret` | pop RIP | 看书签，翻回原来那页 |
| `leave` | mov RSP,RBP; pop RBP | 收拾桌面，恢复之前的布置 |
| `nop` | 什么也不做 | 发呆一秒 |
| `xor a, a` | a = 0（比 mov a,0 快） | 清空容器 |
| `test a, a` | a & a（设标志位，不改变 a） | 照镜子看看自己 |

### 🔥 最值得记住的三条指令的等价实现

```asm
; call func  等价于:
push rip        ; 先保存返回地址
jmp func        ; 再跳转
; 但你不能在汇编里直接写 push rip——CPU 不允许

; ret  等价于:
pop rip         ; 从栈上取出返回地址
; 同样，你不能直接写 pop rip

; leave  等价于:
mov rsp, rbp    ; 把栈顶移到栈底（回收局部变量空间）
pop rbp         ; 恢复调用者的 RBP
```

### 💻 看懂一段真实的汇编

```asm
; C 代码:
; if (x > 10) {
;     printf("big\n");
; } else {
;     printf("small\n");
; }

; 对应汇编（x 在 [rbp-4]）:
    cmp    DWORD PTR [rbp-4], 10    ; 比较 x 和 10
    jle    .L2                       ; 如果 x <= 10，跳到 .L2
    ; x > 10 的分支
    lea    rdi, [str_big]           ; rdi = "big\n" 的地址
    call   puts
    jmp    .L3                       ; 跳过 else 分支
.L2:
    ; x <= 10 的分支
    lea    rdi, [str_small]         ; rdi = "small\n" 的地址
    call   puts
.L3:
    ; if-else 结束，继续执行
```

## 7. 从 C 代码到二进制——编译全流程

### 🏪 生活场景：出版一本书的过程

```
手稿（C 源码）           → 你写的 .c 文件
    ↓ 预处理器（cpp）
去掉批注、展开引用         → .i 文件（纯 C，没有 #include #define）
    ↓ 编译器（cc1）
翻译成汇编               → .s 文件（人类可读的汇编指令）
    ↓ 汇编器（as）
翻译成机器码             → .o 文件（二进制，不可读）
    ↓ 链接器（ld）
合并多个 .o + 库 → 可执行 → a.out / program.exe
```

### 💻 手动走一遍全流程

```bash
cat > hello.c << 'EOF'
#include <stdio.h>
#define MSG "Hello, World!\n"
int main() {
    printf(MSG);
    return 0;
}
EOF

# 1. 预处理（Preprocessing）
gcc -E hello.c -o hello.i
wc -l hello.i       # 可能 800+ 行！stdio.h 被展开了
tail hello.i        # 最后才是你的代码

# 2. 编译（Compilation）—— C → 汇编
gcc -S -masm=intel hello.c -o hello.s
cat hello.s         # 看到了熟悉的 push/mov/call/ret

# 3. 汇编（Assembly）—— 汇编 → 机器码
gcc -c hello.c -o hello.o
xxd hello.o | head  # 全是十六进制字节——这就是 CPU 吃的

# 4. 链接（Linking）—— 合并目标文件
gcc hello.o -o hello
./hello             # Hello, World!

# 5. 看可执行文件的段
objdump -d hello | head -50    # 反汇编
readelf -S hello               # 查看 section 头部
```

### ⚠️ 安全关键：编译选项如何影响漏洞

```bash
# 不同保护级别下的编译
gcc vuln.c -o vuln_no_prot                    # 最弱保护
gcc vuln.c -o vuln_default                    # 默认：NX + ASLR
gcc vuln.c -fstack-protector -o vuln_canary   # 加栈金丝雀
gcc vuln.c -O2 -o vuln_O2                     # 优化可能消除漏洞！

# 攻击实验用（全关）:
gcc -fno-stack-protector -z execstack -no-pie -o vuln vuln.c
```

## 8. 操作系统管的三件事

### 🏪 生活场景：物业公司管理写字楼

一栋写字楼里有很多公司（进程）。物业公司（操作系统）管三件事：

**1. 进程管理 = 排班表**

谁在哪个办公室、用多久。100 家公司要共享 20 间会议室——物业安排时间段，每家轮流用 30 分钟（时间片轮转）。安排得合理，每家都以为自己独占了会议室。

**2. 内存管理 = 分配办公区**

每家公司以为自己租了整层——但实际上一层的空间被分割给不同的公司了。物业通过"虚拟内存"给每家公司一个假象：你有 0x0000 到 0xFFFF 全部地址空间。实际上是共享物理内存。

**3. 文件系统 = 档案管理**

文件存在哪（磁盘上的哪个扇区）、怎么找（目录树→inode→数据块）、谁能看（权限 rwx）。

### 🔍 虚拟内存——最重要的安全概念之一

```
每个进程以为自己独占 4GB（32位）或 128TB（64位）的地址空间：

进程 A 的内存视图:                    进程 B 的内存视图:
┌──────────────────────┐              ┌──────────────────────┐
│ 0x7fffffffffff       │              │ 0x7fffffffffff       │
│       栈              │              │       栈              │
│        ↓              │              │        ↓              │
│                      │              │                      │
│        ↑              │              │        ↑              │
│       堆              │              │       堆              │
│ 0x000000000000       │              │ 0x000000000000       │
└──────────────────────┘              └──────────────────────┘
      (以为自己独占)                        (以为自己独占)

实际物理内存: [A的页] [B的页] [A的页] [空闲] [B的页] ...
                    ↑ 混在一起存放，MMU 负责翻译
```

**安全意义：**
- 每个进程有独立的地址空间——进程 A 不能直接读进程 B 的内存
- **但漏洞可以打破这个隔离：** 内核漏洞 → 读其他进程；共享内存配置错误 → 跨进程泄露
- ASLR（地址空间布局随机化）——每次运行，栈、堆、库的基址随机偏移，让攻击者猜不到地址
- 32 位 ASLR 因为地址空间小（4GB），随机性不够，相对容易被暴力破解

## 9. 🧪 动手：写一个 CPU 模拟器

> 这个实验让你真正理解"CPU 就是不断取指令→译码→执行的循环"。

```python
# cpu_sim.py —— 一个极简的 CPU 模拟器
# 支持的指令: mov reg, val / add reg, reg / jmp addr / cmp / je / hlt

class TinyCPU:
    def __init__(self):
        self.regs = {'A': 0, 'B': 0, 'C': 0}  # 寄存器
        self.flags = {'zero': False}           # 标志位
        self.mem = [0] * 256                    # 内存 256 字节
        self.pc = 0                             # 程序计数器（=RIP）
        self.running = False

    def load(self, program):
        """把程序加载到内存"""
        for i, byte in enumerate(program):
            self.mem[i] = byte

    def fetch(self):
        """取指：从 PC 指向的地址读一个字节"""
        opcode = self.mem[self.pc]
        self.pc += 1
        return opcode

    def run(self):
        self.running = True
        while self.running:
            op = self.fetch()

            if op == 0x01:       # mov reg, val
                reg = self.fetch()
                val = self.fetch()
                name = chr(ord('A') + reg)
                self.regs[name] = val
                print(f"  mov {name}, {val}")

            elif op == 0x02:     # add regA, regB
                ra = chr(ord('A') + self.fetch())
                rb = chr(ord('A') + self.fetch())
                self.regs[ra] += self.regs[rb]
                print(f"  add {ra}, {rb}  → {ra}={self.regs[ra]}")

            elif op == 0x03:     # cmp regA, regB
                ra = chr(ord('A') + self.fetch())
                rb = chr(ord('A') + self.fetch())
                self.flags['zero'] = (self.regs[ra] == self.regs[rb])
                print(f"  cmp {ra},{rb}  zero={self.flags['zero']}")

            elif op == 0x04:     # je addr (相等就跳)
                addr = self.fetch()
                if self.flags['zero']:
                    self.pc = addr
                    print(f"  je {addr}  ← 跳了！")

            elif op == 0x05:     # jmp addr (无条件跳)
                addr = self.fetch()
                self.pc = addr
                print(f"  jmp {addr}")

            elif op == 0xFF:     # hlt (停机)
                print("  hlt — 程序结束")
                self.running = False

        print(f"\n最终寄存器状态: {self.regs}")

# === 运行一个测试程序 ===
# 程序: A=5; B=3; A=A+B; 如果 A==8 就跳到末尾; B=99(应该被跳过); 结束
program = [
    0x01, 0, 5,    # mov A, 5
    0x01, 1, 3,    # mov B, 3
    0x02, 0, 1,    # add A, B  → A=8
    0x03, 0, 1,    # cmp A, B  → not equal
    0x04, 14,      # je 14 (如果相等就跳到地址14)
    0x01, 1, 99,   # mov B, 99 (没跳，B=99)
    0xFF,          # hlt
]
cpu = TinyCPU()
cpu.load(program)
print("=== 程序执行追踪 ===\n")
cpu.run()
# 预期: A=8, B=99（没有跳，因为 A(8) != B(3)）
```
