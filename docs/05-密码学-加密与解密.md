## 1. 📌 对称加密——用同一把钥匙

### 🏪 生活场景：你家的防盗门

你家大门用的是一把钥匙。**出门时用这把钥匙锁门，回家时用同一把钥匙开门。** 这就是对称加密——加密和解密用同一把密钥。

```
加密: 明文 + 密钥 → 密文
解密: 密文 + 密钥 → 明文

出门: 家里财物(明文) + 钥匙(密钥) → 锁上的门(密文)
回家: 锁上的门(密文) + 钥匙(密钥) → 家里财物(明文)
```

**核心问题：钥匙怎么给对方？** 如果你们面对面，你可以直接把钥匙交给他。但在互联网上，你怎么"递"一把钥匙给一个远在千里之外的人？发微信？发邮件？——这些渠道本身就不安全，如果有人偷看，钥匙就泄露了。

这就是为什么对称加密不能单独用于互联网通信——你需要一个安全的"钥匙传递"方案。这是 §2 非对称加密要解决的问题。

### 🔍 AES——现代对称加密的标准

AES（Advanced Encryption Standard，高级加密标准）是由比利时密码学家设计的，经过全球专家 5 年公开竞赛评选出来的。**你手机、电脑、路由器、银行卡里的加密，几乎都是 AES。**

AES 的基本参数：

| 参数 | AES-128 | AES-256 |
|------|---------|---------|
| 密钥长度 | 16 字节（128 位） | 32 字节（256 位） |
| 暴力破解难度 | 2^128 次尝试 | 2^256 次尝试 |
| 破解需要时间 | 即使全世界的计算机一起算，也需要数十亿年 | 比 128 位更难 |

**AES 不是靠"算法保密"来保证安全的——算法是公开的，安全性完全依赖密钥。** 全世界都知道你家门锁的原理（锁芯怎么转），但只有你有那把钥匙。

### 🔍 分组密码与工作模式

AES 是分组密码——每次处理 16 字节（128 位）的数据块。如果数据超过 16 字节，需要决定"怎么把多个块连起来"。这就是**工作模式**。

```
ECB 模式（电子密码本）——千万不要用！
┌─────────────────────────────────────────────┐
│ 原始图片（一个企鹅的像素）                     │
│  ┌────┬────┐                                │
│  │ ☀  │ ☀  │ ← 两个块内容相同                 │
│  │ ☀  │ ☁  │                                │
│  └────┴────┘                                │
│                                             │
│ ECB 加密后:                                  │
│  ┌────┬────┐                                │
│  │ @# │ @# │ ← 相同的块→相同的密文！            │
│  │ @# │ $% │   企鹅的轮廓还能看出来！            │
│  └────┴────┘                                │
│                                             │
│ → 这是一个真实存在的问题！                      │
│   有人用 ECB 模式加密过企鹅图片，加密后轮廓清晰可见  │
└─────────────────────────────────────────────┘

CBC 模式（密码块链接）——比 ECB 好：
  每个明文块先跟前一个密文块 XOR，再加密。
  第一个块没有"前一个密文块"，所以用一个随机 IV（初始化向量）。
  即使两个块内容一样，密文也不同。

CTR/GCM 模式——现代推荐：
  GCM 除了加密，还提供"完整性验证"——
  如果密文在传输中被改了，解密时能发现。
```

### 💻 Python 实现 AES 加密/解密（推荐用 cryptography 库）

```python
# pip install cryptography
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.backends import default_backend
import os

# 生成随机的 32 字节密钥（AES-256）
key = os.urandom(32)
# 生成随机的 16 字节 IV（用于 CBC 模式）
iv = os.urandom(16)

def aes_encrypt(plaintext: bytes, key: bytes, iv: bytes) -> bytes:
    cipher = Cipher(algorithms.AES(key), modes.CBC(iv), backend=default_backend())
    encryptor = cipher.encryptor()

    # AES-CBC 要求数据长度是 16 的倍数，需要填充
    pad_len = 16 - (len(plaintext) % 16)
    padded = plaintext + bytes([pad_len] * pad_len)  # PKCS7 填充
    return encryptor.update(padded) + encryptor.finalize()

def aes_decrypt(ciphertext: bytes, key: bytes, iv: bytes) -> bytes:
    cipher = Cipher(algorithms.AES(key), modes.CBC(iv), backend=default_backend())
    decryptor = cipher.decryptor()
    padded = decryptor.update(ciphertext) + decryptor.finalize()

    # 去掉 PKCS7 填充
    pad_len = padded[-1]
    return padded[:-pad_len]

# === 测试 ===
message = b"转账 100 元给小明"
print(f"原文: {message}")

encrypted = aes_encrypt(message, key, iv)
print(f"密文: {encrypted.hex()}")

decrypted = aes_decrypt(encrypted, key, iv)
print(f"解密: {decrypted}")

# 如果用错误的密钥？试试看！
wrong_key = os.urandom(32)
try:
    wrong_result = aes_decrypt(encrypted, wrong_key, iv)
    print(f"用错密钥的结果: {wrong_result}")  # 乱码，或者解密失败
except:
    print("用错密钥 → 解密失败或得到乱码")
```

## 2. 📌 非对称加密——公钥和私钥

### 🏪 生活场景：街头的邮政信箱

对称加密的问题：你得先把钥匙给对方。但怎么安全地把钥匙递过去呢？

**非对称加密的解法：**

街角有一个邮政信箱。信箱上有一条缝——任何人都可以往里面投信，不需要钥匙。但只有邮递员有一把钥匙能打开信箱取信。

```
公钥 = 信箱的投递口
  → 公开的、谁都可以拿到
  → 只能用来加密（投信进去）
  → 不能用来解密（信投进去了就拿不出来了）

私钥 = 邮递员的钥匙
  → 绝对保密、只有一个人有
  → 能打开投进去的信（解密）
```

**所以在互联网上：**
1. 百度把公钥公开在网站上："这是我的公钥，大家随便拿！"
2. 你用百度的公钥加密你的信息，发给百度
3. 只有百度有私钥，所以只有百度能解密
4. 中间人即使截获了密文和公钥，也解不开——公钥只能加密不能解密！

### 🔍 RSA 的数学原理——两句话版

**RSA 的安全性基于一个事实：两个大质数相乘很容易，但把乘积分解回两个质数极其困难。**

```
比如：我给你 3233，让你分解成两个质数的乘积。
你可以试出来：3233 = 61 × 53（5 分钟）

但如果我给你一个 600 位的数字？
全世界所有的计算机一起算，需要的时间比宇宙的年龄还长。
```

### 🔍 RSA 密钥生成与加解密流程

```
生成密钥对：
  ① 选两个大质数 p 和 q（实际程序用几百位的质数）
  ② n = p × q               （n 是公开的）
  ③ φ(n) = (p-1) × (q-1)   （φ(n) 是秘密的——因为需要知道 p 和 q）
  ④ 选 e = 65537            （常用的公钥指数，素数，够大）
  ⑤ 算 d = e⁻¹ mod φ(n)    （私钥指数——只有知道 φ(n) 才能算出来）

公钥 = (e, n)    "用 e 和 n 加密"
私钥 = (d, n)    "用 d 和 n 解密"

加密: 密文 = (明文 ^ e) mod n
解密: 明文 = (密文 ^ d) mod n

验证: (明文 ^ e) ^ d = 明文 ^ (e×d) = 明文 ^ 1 = 明文  (mod n)
      因为 e × d ≡ 1 (mod φ(n))，根据欧拉定理成立
```

### 💻 Python 实现 RSA（教学版——不要在生产环境用！）

```python
import math

# ===== 生成密钥（用很小的质数来演示） =====
p, q = 61, 53                    # 实际要用几百位的质数！
n = p * q                         # n = 3233
phi = (p - 1) * (q - 1)          # phi = 3120
e = 17                            # 公钥指数

# 计算私钥 d：d = e 在 mod phi 下的乘法逆元
# 即 e × d ≡ 1 (mod phi)
d = pow(e, -1, phi)               # Python 3.8+ 支持 pow(x, -1, mod)
print(f"公钥: (e={e}, n={n})")
print(f"私钥: (d={d}, n={n})")
# 输出: 公钥: (e=17, n=3233)
#       私钥: (d=2753, n=3233)

# ===== 加密 =====
message = 123                     # 实际消息要分块，每个块 < n
ciphertext = pow(message, e, n)   # 123^17 mod 3233
print(f"原文: {message} → 密文: {ciphertext}")

# ===== 解密 =====
decrypted = pow(ciphertext, d, n)  # cipher^2753 mod 3233
print(f"密文: {ciphertext} → 解密: {decrypted}")

# ===== 验证：用公钥加密的信息不能用公钥解密 =====
try_wrong = pow(ciphertext, e, n)
print(f"用公钥尝试解密: {try_wrong}  ← 不是原文！失败了！")
```

### 🔥 为什么 HTTPS 用混合加密？

```
纯用非对称加密（RSA）加密所有数据:
  RSA 比 AES 慢大约 1000-10000 倍
  一个网页 100KB，用 RSA 加密 → 加载要好几秒

混合加密（实际做法）:
  ① 握手阶段: 用 RSA/ECDH 安全传输一个"会话密钥"（几十字节就够了）
  ② 数据传输: 用 AES + 会话密钥加密所有数据（极快）

这样既解决了"钥匙怎么给对方"的问题，
又不会因为非对称加密太慢而卡死。
```

## 3. 📌 哈希函数——数据的指纹

### 🏪 生活场景：文件的电子指纹

你去银行办业务，柜员要核实你的身份。你掏出身份证，柜员拿机器一扫，系统里显示出你的预留指纹——比对通过。

**哈希函数就是给数据生成"指纹"。** 你把任意长度的一段数据喂给它，它吐出一段固定长度的"摘要"（hash值）。

```
文件: 1GB 的电影 → SHA-256 → 64 个十六进制字符
文件: "hello"  → SHA-256 → 64 个十六进制字符
同一个文件，每次哈希结果一样
修改一个比特，结果面目全非
```

### 🔍 哈希的五个核心特性

```
① 固定长度输出   → 输入可以 1 字节，也可以 1GB，输出长度不变
② 单向性         → 有哈希值算不出原文
③ 雪崩效应       → 改输入 1 bit，输出彻底不同
④ 抗碰撞         → 找不到两个不同输入有相同的哈希（理想情况）
⑤ 快速计算       → 比加密算法快得多
```

### 🔍 常见哈希算法对比

| 算法 | 输出 | 安全性 | 速度 | 用途 |
|------|:---:|:-----:|:---:|------|
| MD5 | 128bit | ❌ 已破解 | 极快 | 仅用于校验非安全场景 |
| SHA-1 | 160bit | ❌ 已破解 | 快 | Git 仍用但已准备迁移 |
| SHA-256 | 256bit | ✅ 安全 | 较快 | 当前主流，密码存储/区块链 |
| SHA-512 | 512bit | ✅ 安全 | 较快 | 更高安全需求 |
| bcrypt | 可变 | ✅ 安全 | **慢** | 专为密码存储设计 |

### 💻 动手感受"雪崩效应"

```python
import hashlib

def show_hash(label, data):
    h = hashlib.sha256(data).hexdigest()
    print(f"{label}: {h}")

show_hash("hello", b"hello")
show_hash("Hello", b"Hello")    # 首字母大写
show_hash("hellp", b"hellp")    # 改最后一个字母
show_hash("hello.", b"hello.")  # 加一个点
```

输出：

```
hello:  2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
Hello:  185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
hellp:  b6e6f7aa751f613d0c2c3aff0fdedaeb6e6f4d3bf9d7b6eef1016e96a1f5bd53
hello.: d42e59630dfbf47e32be95eaae4acd1c0f01cfd0e5d21cef854d59e5452d3a23
```

**四个哈希值完全不同！改一个 bit，整个指纹变了。** 这就是雪崩效应。

### 🔍 网安中的哈希应用

```
① 密码存储: 数据库不存 "password123"，存 bcrypt(password123)
   即使数据库泄露，攻击者也拿不到密码原文

② 文件校验: 下载 Kali Linux 的 ISO 后，算 SHA256，
   跟官网公布的对比——确保文件没有被篡改过

③ 数字签名: 因为哈希比原文短得多，
   所以签名时先哈希再对摘要签名，高效

④ 区块链: 每个区块包含前一个区块的哈希，
   改了任何一个区块 → 后面所有区块的哈希全变 → 立即被发现

⑤ 取证: 对硬盘镜像做哈希 → 生成唯一指纹
   → 证明数据在分析过程中没有被改动
```

## 4. 数字签名与证书——互联网信任体系

### 🏪 生活场景：签合同按手印

你要签一份合同。你在签名处按了一个手印。

```
签名过程: 合同内容 → 你在上面按手印 → 签字完成
验证过程: 谁拿到这份合同，都可以拿你的公开指纹样本比对这个手印
         → 一致 = 确认是你签的、内容没被改过
         → 不一致 = 要么不是你签的、要么合同被人掉了包
```

**数字签名 = 合同内容（哈希）+ 你的手印（用私钥加密哈希值）**

### 🔍 签名的完整流程

```
发送方（你）签名:
  ┌──────────┐     哈希      ┌──────────┐    用私钥加密    ┌──────────┐
  │ 原文件    │ ──────────→ │ 摘要(短)   │ ────────────→ │  签名     │
  └──────────┘              └──────────┘                └──────────┘
  发送给接收方: [原文件] + [签名]

接收方验证:
  ┌──────────┐     哈希      ┌──────────┐
  │ 原文件    │ ──────────→ │ 摘要1     │
  └──────────┘              └──────────┘     比较 摘要1 == 摘要2？
                                              ↓
  ┌──────────┐   用公钥解密    ┌──────────┐    相等 = 验证通过！
  │ 签名      │ ──────────→ │ 摘要2     │    不等 = 被篡改或假冒
  └──────────┘              └──────────┘
```

**签名同时实现了两个功能：**
- 身份认证：私钥只有你有 → 签名证明"这个东西是你发的"
- 完整性验证：内容改了哈希就对不上 → 签名证明"内容没人动过"

### 🔍 证书链——"我凭什么相信这个公钥是你的？"

一个关键问题：你怎么知道你拿到的公钥真的是百度的，而不是中间人伪造的？

比如你在咖啡厅连了 WiFi。中间人拦截了你的请求，说"我就是百度，这是我的公钥！"——他给的是他自己的公钥。你用它加密了你的密码发过去，他用他的私钥解密，看到了你的密码。

**数字证书解决这个问题。**

```
根 CA 证书（预装在你手机/电脑里，操作系统出厂时就信任）
  │  比如: DigiCert, Let's Encrypt, GlobalSign
  │
  └→ 签发了 中间 CA 证书
        │  "我验证过这家公司，它的公钥是真的"
        │
        └→ 签发了 baidu.com 的 SSL 证书
              "百度公司的 baidu.com 域名的公钥是 XXXX"

浏览器拿到 baidu.com 的证书后:
  → 看是谁签发的 → 中间 CA
  → 中间 CA 是谁签发的 → 根 CA
  → 根 CA 在操作系统的信任列表里 → 信任！
```

**网安意义：**
- 如果攻击者在你电脑上装了自签名的根证书 → 他就能伪造任何网站的证书 → 中间人解密所有 HTTPS 流量
- 这就是为什么 BurpSuite 抓 HTTPS 包需要你在浏览器里安装它的 CA 证书

## 5. 常见密码攻击

| 攻击 | 原理 | 生活类比 |
|------|------|---------|
| 暴力破解 | 试所有可能的密钥 | 0000→0001→...→9999 试行李箱密码 |
| 字典攻击 | 用常见密码列表试 | 用 "password123,admin,123456..." 挨个尝试 |
| 彩虹表 | 预计算大量哈希→原文的对照表 | 考试前把答案抄好，考场上直接查表 |
| 频率分析 | 密文中出现次数最多的字符可能对应原文高频字母 | 英文里 e 出现最频繁——看谁最频繁 |
| 中间人攻击 | 截获通信，对双方各冒充对方 | 假快递员在你和卖家之间传话，两边都骗 |
| 重放攻击 | 截获合法请求，重复发送 | 录下你开门禁的"嘀"，再播一次 |
| 边信道攻击 | 不破算法，利用功耗/时间/电磁泄露 | 听保险柜转盘的声音判断密码 |
| 哈希碰撞 | 找到两个不同输入有相同的哈希值 | 伪造一份合同，哈希值和原合同一样 |

### 🔥 密码学的两条黄金法则

> **1. 永远不要自己发明加密算法。** 你以为加了点"花样"就更安全——实际上只是给自己制造了一种虚假的安全感。用 AES-GCM、Curve25519、SHA-256。这些算法经全球密码学家分析了几十年。

> **2. 实现正确 ≠ 算法安全。** 你用对了 AES-GCM，但密钥写在代码里上传了 GitHub——等于裸奔。算法是安全的，人是漏洞。

## 6. 🧪 Python 密码学综合实验

### 实验 1：感受 RSA 的"慢"

```python
import time
from cryptography.hazmat.primitives.asymmetric import rsa
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
import os

encryptor = cipher.encryptor()

start = time.time()
# PKCS7 填充
pad_len = 16 - (len(data) % 16)
padded = data + bytes([pad_len] * pad_len)
encryptor.update(padded)
encryptor.finalize()
print(f"AES-256 加密 1MB 数据耗时: {time.time() - start:.4f} 秒")
print("→ AES 比 RSA 快几百到上千倍")

import os
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes

# 1. 服务器生成 RSA 密钥对
print("=== 模拟 HTTPS 握手 ===")
server_private = rsa.generate_private_key(65537, 2048)
server_public = server_private.public_key()

# 2. 客户端生成会话密钥（AES 密钥）
session_key = os.urandom(32)  # AES-256 密钥
print(f"客户端生成会话密钥: {session_key.hex()[:32]}...")

# 3. 用服务器公钥加密会话密钥 → 传给服务器
encrypted_key = server_public.encrypt(
    session_key,
    padding.OAEP(mgf=padding.MGF1(hashes.SHA256()), algorithm=hashes.SHA256(), label=None)
)
print(f"用 RSA 公钥加密会话密钥: {encrypted_key.hex()[:64]}...")

# 4. 服务器用私钥解密 → 得到会话密钥
decrypted_key = server_private.decrypt(
    encrypted_key,
    padding.OAEP(mgf=padding.MGF1(hashes.SHA256()), algorithm=hashes.SHA256(), label=None)
)
print(f"服务器解密得到会话密钥: {decrypted_key.hex()[:32]}...")
print(f"密钥匹配: {session_key == decrypted_key}")

# 5. 之后用 AES + 会话密钥通信
message = b"用户支付密码的哈希值"
iv = os.urandom(16)
cipher = Cipher(algorithms.AES(session_key), modes.CBC(iv))
encryptor = cipher.encryptor()
# ... 后续通信都用 AES + session_key
```
