## 1. 📌 分层模型——为什么网络要分层？

### 🏪 生活场景：寄一个快递包裹

你在淘宝上卖了一本书，要去快递站寄给买家。整个过程天然是分层的：

```
第 5 层（应用层）：你写了一封感谢信夹在书里
    → 发货人关心的是"信的内容"。快递员不需要知道信里写了什么。

第 4 层（传输层）：你把书和信装进纸箱，快递员贴上快递单
    → 快递单上有单号，快递公司用它来追踪包裹是否丢失。如果丢了，凭单号重发。

第 3 层（网络层）：快递站的分拣中心看收件地址"广东省深圳市……"
    → 分拣员只看城市和区，决定包裹走陆运还是空运——不管纸箱里是书还是手机。

第 2 层（链路层）：货车司机拉着你的包裹在高速上跑
    → 司机只负责把货从 A 点运到 B 点，不关心包裹里是什么、收件人是谁。

第 1 层（物理层）：高速公路的路面、车道线、信号灯
    → 这是"物理基础"，所有车辆都在上面跑。
```

**每一层只对上层的"内容"负责，不管下层的"运输细节"。** 快递分拣员不需要知道书里写了什么。网络路由器不需要知道你发的 HTTP 请求里是什么 URL。

### 🔍 TCP/IP 四层模型（实际使用的）

```
┌─────────────────────────────────────────────────────────────┐
│  层             做什么                  数据单元      类比      │
├─────────────────────────────────────────────────────────────┤
│  应用层         你写的程序              数据          信的内容  │
│  HTTP/FTP/DNS   浏览器/微信/邮件                            │
├─────────────────────────────────────────────────────────────┤
│  传输层         端口号 + 可靠性          段(Segment)   快递单  │
│  TCP/UDP        TCP保证不丢件                              │
├─────────────────────────────────────────────────────────────┤
│  网络层         IP地址 + 路由            包(Packet)    地址标签│
│  IP             全球寻址                                  │
├─────────────────────────────────────────────────────────────┤
│  网络接口层      MAC地址 + 电信号         帧(Frame)     货车    │
│  Ethernet/WiFi  局域网传输                                │
└─────────────────────────────────────────────────────────────┘
```

### 🔍 数据封装过程——一层套一层

你要发送 "Hello" 给 `www.baidu.com` 的 80 端口：

```
应用层:    [GET / HTTP/1.1                   ]  ← 浏览器生成
              ↓ 交给传输层："帮我可靠地发给 80 端口"
传输层:    [TCP头: 源端口54321→目的端口80    | GET /... ]  ← 加了端口
              ↓ 交给网络层："帮我发到 110.242.68.66"
网络层:   [IP头: 源IP→110.242.68.66         | TCP头 | GET /... ]
              ↓ 交给链路层："帮我从这块网卡发出去"
链路层:  [MAC头: 源MAC→网关MAC              | IP头 | TCP头 | GET | FCS校验]
              ↓ 网卡转成电信号/光信号
物理层:  ~~~~~~~~ 0101010100101 ~~~~~~~~→  网线/光纤/WiFi
```

**接收方一层层剥皮：** 链路层剥 MAC 头 → 网络层剥 IP 头 → 传输层剥 TCP 头 → 应用层得到原始 HTTP 请求。

### 💻 动手看：你的数据包长什么样

```bash
# 用 curl 发一个请求，同时看详细过程
curl -v http://httpbin.org/get
# -v (verbose) 会打印：
# * Trying 3.94.158.129:80...          ← 网络层：DNS→IP
# * Connected to httpbin.org            ← 传输层：TCP 连接建立
# > GET /get HTTP/1.1                   ← 应用层：HTTP 请求
# > Host: httpbin.org
# > User-Agent: curl/8.0.1
# ...
# < HTTP/1.1 200 OK                     ← 应用层：HTTP 响应
```

## 2. 📌 TCP——可靠传输的秘密

### 🏪 生活场景：打电话预约餐厅

你打电话给一家餐厅预约晚餐：

```
你: "喂？你好，请问是 XX 餐厅吗？"              ← SYN
餐厅: "是的，请问有什么可以帮您？"              ← SYN+ACK
你: "我想预约今晚 7 点，3 个人。"                ← ACK + 数据
餐厅: "好的，已登记。晚上见。"                  ← ACK + 数据
你: "好的，再见。"                              ← FIN
餐厅: "好的再见。"                              ← FIN+ACK
你: "拜。"                                      ← ACK
```

这就是 TCP 连接的全过程——**三次握手建连接 + 数据传输 + 四次挥手断连接。**

每一步对方都确认"收到了"（ACK）。如果你说了一句对方没应答，你会再问一遍"你听到了吗？"（超时重传）。这就是"可靠传输"——TCP 保证数据不丢、不乱序。

### 🔍 三次握手（建立连接）

```
客户端                                          服务器
   │                                               │
   │── SYN (seq=x, flag=SYN) ──────────────────→│  ① "在吗？"
   │   客户端随机选一个起始序号 x                    │     SYN = Synchronize
   │                                               │
   │←─ SYN+ACK (seq=y, ack=x+1, flag=SYN│ACK) ──│  ② "在的，收到你的请求"
   │   服务器选自己的起始序号 y                      │     ACK = Acknowledge
   │   ack=x+1 表示"x 之前的我都收到了"              │
   │                                               │
   │── ACK (ack=y+1, flag=ACK) ────────────────→│  ③ "好的，开始传数据吧"
   │   客户端确认收到了服务器的起始序号              │     此时连接建立（ESTABLISHED）
   │                                               │
   │<═══════ 数据传输（PSH, ACK） ════════>      │
```

**为什么是三次而不是两次？** 假设网络很差，客户端发了一个 SYN，卡在路上 30 秒。客户端以为丢了，又发了一个新的 SYN。然后旧的 SYN 突然到了服务器——如果是两次握手，服务器直接就"连接建立"了，等着收数据。但客户端根本不知道这个旧连接的存在！服务器的资源被浪费了。

**为什么不是四次？** 因为服务端的 SYN 和 ACK 可以合并在一起发——SYN+ACK 一个包就够了。

**网安意义：SYN Flood 攻击。** 攻击者发送大量 SYN，但不回应最后的 ACK。服务器为每个半连接分配资源，最终资源耗尽，无法接受新连接。——这就是 DDoS 攻击中最经典的一种。

### 🔍 四次挥手（断开连接）

```
客户端                                          服务器
   │                                               │
   │── FIN (flag=FIN│ACK) ────────────────────→│  ① "我说完了"
   │   客户端没有数据要发了                        │
   │                                               │
   │←─ ACK ───────────────────────────────────│  ② "知道了"
   │   服务器确认收到了 FIN                        │  (但服务器可能还有数据要发)
   │                                               │
   │   ... 服务器继续发剩余数据 ...                  │
   │                                               │
   │←─ FIN (flag=FIN│ACK) ────────────────────│  ③ "我也说完了"
   │   服务器也没有数据要发了                       │
   │                                               │
   │── ACK ──────────────────────────────────→│  ④ "好的，拜拜"
   │   客户端进入 TIME_WAIT，等 2MSL（约60秒）       │
   │   防止最后的 ACK 丢失 + 让旧连接的包消失        │
   │   60 秒后完全关闭                              │
```

**为什么是四次不是三次？** 因为 TCP 是全双工的——双方都可以独立决定"我不发了"。服务器收到 FIN 后可能还有数据要传，所以 ACK 和 FIN 必须分开。

### 🔍 TCP 的可靠性机制

```
确认应答 (ACK):     每收到一个包，回复 ACK"收到了"
超时重传:           发出去 N 秒没收到 ACK → 自动重发
流量控制 (滑动窗口): 接收方告诉发送方"我还能接收多少"
                      → 防止发送方把接收方"淹死"
拥塞控制:           如果丢包率高，自动降低发送速率
                      → 所有人都慢一点，避免网络拥堵
```

### 💻 动手看 TCP 状态

```bash
# 看本机所有 TCP 连接

# 过滤看 TIME_WAIT 状态（四次挥手最后阶段）
netstat -an | grep TIME_WAIT
# 每个连接关闭后会在 TIME_WAIT 状态待约 60 秒

# 手动建立和关闭一个 TCP 连接

                         # 按两次回车

```

## 3. UDP 与 ICMP——不需要握手的协议

### 🏪 生活场景：发微信消息 vs 寄挂号信

**UDP = 发微信消息。** 发出去就发出去了——不确认对方收到没，不关心顺序。一句话丢了就丢了。**适用于：视频通话（丢几帧无所谓）、游戏（速度更重要）、DNS 查询（一个包就能完成）。

**TCP = 寄挂号信。** 要回执、要签收、要追踪。丢了重新寄。**适用于：网页、邮件、文件传输——丢一个字节都不行。

### 🔍 UDP 头——极简设计

```
UDP 头只有 8 字节：
┌──────────────┬──────────────┬──────────────┬──────────────┐
│ 源端口 (2B)   │ 目的端口 (2B) │ 长度 (2B)    │ 校验和 (2B)   │
└──────────────┴──────────────┴──────────────┴──────────────┘
然后直接跟数据。

TCP 头要 20-60 字节，UDP 只要 8 字节 —— 因为什么都没管。
```

### 🔍 ICMP——网络的诊断工具

ping 和 traceroute 用的就是 ICMP。它不是用来传数据的，是用来"问路"和"测速"的。

```bash
# ping: 发 ICMP Echo Request，对方回 Echo Reply
ping -c 4 www.baidu.com
# PING www.baidu.com (110.242.68.66): 56 data bytes
# 64 bytes from 110.242.68.66: icmp_seq=0 ttl=52 time=38.5 ms
# ttl 是"还能经过多少个路由器"——每过一个路由器减 1

# traceroute: 看数据包经过了哪些路由器
traceroute www.baidu.com       # Linux/Mac
tracert www.baidu.com          # Windows
# 输出每一跳的 IP 和耗时
```

**网安意义：**
- 渗透测试信息收集阶段用 ping 判断目标是否在线
- 用 traceroute 摸清网络拓扑
- 某些防火墙禁 ICMP——可以用来探测防火墙策略

## 4. 📌 HTTP——Web 的通用语言

### 🏪 生活场景：在餐厅扫码点餐

你去一家餐厅，桌上贴着二维码。整个过程就是一次 HTTP 交互：

```
① 扫码 → 浏览器 GET 请求菜单页
  GET /menu?table_id=12 HTTP/1.1
  Host: order.restaurant.com

② 服务员（服务器）端来菜单 → HTTP 响应
  HTTP/1.1 200 OK
  Content-Type: text/html
  [菜单页 HTML]

③ 你选好了菜点"下单" → POST 请求
  POST /order HTTP/1.1
  Content-Type: application/json
  {"table":12, "items":[{"name":"宫保鸡丁","qty":1}]}

④ 厨房收到订单 → 200 OK
  {"order_id": 8842, "status": "cooking", "eta": "15min"}

⑤ 吃完饭你点"结账" → GET 请求
  GET /bill?order_id=8842 HTTP/1.1

⑥ 账单返回 → 你扫码支付（这是另一套流程了，涉及支付 API）
```

这 6 步就是 Web 应用的全部。每个网页、每个 App、每个 API 都在做这件事：**客户端发请求 → 服务器处理 → 返回结果。**

### 🔍 HTTP 请求格式（逐字节解读）

```
GET /api/products?id=123 HTTP/1.1       ← 请求行: 方法 URL 版本
Host: shop.example.com                  ← 请求头(Headers)
User-Agent: Mozilla/5.0 (Windows NT 10.0)
Accept: application/json
Accept-Language: zh-CN,zh;q=0.9
Cookie: session_id=abc123xyz
Authorization: Bearer eyJhbGc...
Referer: https://shop.example.com/
                                       ← 空行！表示头部结束
                                       ← (POST 请求这里放 Body)
```

**安全人必须看懂的每个 Header：**

| Header | 干什么的 | 安全意义 |
|--------|---------|---------|
| Host | 告诉服务器我要访问哪个网站 | Host 头攻击：改 Host 可能访问到其他虚拟主机 |
| User-Agent | 我用什么浏览器 | 可以伪造绕过弱智的 UA 检查 |
| Cookie | 我的身份凭证 | ★ 偷了 Cookie = 偷了身份 |
| Authorization | Bearer Token/API Key | 信息泄露——Token 存前端可能被 XSS 偷 |
| Referer | 我从哪个页面跳过来的 | CSRF 防御时检查 Referer |
| Content-Type | Body 是什么格式 | 如果服务器只检查这个来判断文件类型→文件上传绕过 |
| X-Forwarded-For | 原始客户端 IP（代理加的） | 可以伪造绕过 IP 限制 |

### 🔍 HTTP 响应格式

```
HTTP/1.1 200 OK                          ← 状态行
Server: nginx/1.24.0                     ← 服务器指纹！
Date: Mon, 03 Jun 2026 10:30:00 GMT
Content-Type: application/json
Content-Length: 156
Set-Cookie: session=xyz789; HttpOnly; Secure; SameSite=Strict
                                         ← 空行
{"status":"success", "data": [...]}       ← 响应体
```

### 🔍 状态码速查（安全方向）

| 状态码 | 含义 | 安全场景 |
|:------:|------|---------|
| 200 | 正常 | 爆破成功/注入有结果 |
| 301/302 | 重定向 | 开放重定向漏洞（redirect 到恶意站）|
| 401 | 没认证 | 访问需要登录的页面 |
| 403 | 没权限 | 目录爆破中区分"存在但禁止"和"不存在"的关键 |
| 404 | 不存在 | 目录爆破的基础判断 |
| 500 | 服务器错误 | ★ SQL 注入导致数据库报错会出现 500 |

### 💻 用 Python 手动构造各种 HTTP 请求

```python
import requests

# === GET: 获取信息 ===
r = requests.get("http://httpbin.org/get", params={"page": "1", "limit": "10"})

# === POST: 提交数据 ===
# 表单格式
r = requests.post("http://httpbin.org/post", data={"user": "admin", "pass": "123"})
# JSON 格式
r = requests.post("http://httpbin.org/post", json={"user": "admin"})

# === PUT: 更新资源 ===
r = requests.put("http://httpbin.org/put", data={"key": "new_value"})

# === DELETE: 删除 ===
r = requests.delete("http://httpbin.org/delete")

# === 模拟各种 HTTP 方法 ===
for method in ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS', 'HEAD']:
    r = requests.request(method, f"http://httpbin.org/{method.lower()}")
    print(f"{method}: {r.status_code}")
```

## 5. HTTPS 与 TLS——加密传输

### 🏪 生活场景：明信片 vs 密码锁快递箱

**HTTP = 明信片。** 你写了什么，途中每个经手的人（路由器管理员、WiFi 所有者、ISP 员工）都能看到。

**HTTPS = 带密码锁的快递箱。** 你上锁（加密），只有收件人知道密码。途中的快递员只能看到一个箱子，打不开。

### 🔍 TLS 握手的核心过程

```
客户端                                          服务器
   │                                               │
   │── ClientHello ────────────────────────────→│  ① "你好，我支持的加密方式有
   │   (TLS版本 + 支持的密码套件 + 随机数1)          │     AES-GCM, ECDHE-RSA..."
   │                                               │
   │←─ ServerHello + 证书 ────────────────────│  ② "好，我们用 ECDHE-RSA"
   │   (选定的密码套件 + 随机数2 + ★服务器证书)       │    亮出我的身份证（证书）
   │                                               │
   │── 验证证书 ──                                 │  ③ 浏览器检查:
   │   1. 证书是受信任的 CA 发的吗？                  │     证书有效性 ✓
   │   2. 证书的域名和我要访问的一致吗？               │     域名匹配 ✓
   │   3. 证书过期了吗？                             │     在有效期内 ✓
   │   4. CA 签名的有效性                             │     签名验证通过 ✓
   │                                               │
   │── 密钥交换 + 生成会话密钥 ────────────────→│  ④ 双方各自算出相同的
   │   (用服务器公钥加密预主密钥)                     │     "会话密钥"
   │                                               │
   │   ←══ 此后所有通信都用会话密钥对称加密 ═══→   │
   │         (AES-GCM，速度很快)                    │
```

**网安意义：** 渗透测试中分析 HTTPS 流量需要做中间人解密——通过 BurpSuite 的代理证书把自己伪装成"可信的中间人"。

## 6. 📌 DNS 与 ARP——网络世界的电话本

### 🏪 生活场景：打 114 查号台

你想给"百度公司"打电话。你不知道他们的号码。于是你打 114（DNS 服务器）："帮我查一下'百度公司'的电话号码。"114 查号台告诉你："110.242.68.66。"——然后你拨号过去。

**DNS = 互联网的 114 查号台。** 你把域名（www.baidu.com）给它，它把 IP 地址（110.242.68.66）返给你。

### 🔍 DNS 查询的完整过程（你的浏览器输入 baidu.com 后）

```
你的电脑: "www.baidu.com 的 IP 是什么？"
    ↓ ① 先查自己的 hosts 文件（/etc/hosts）
    ↓ 没找到 ↓
    ↓ ② 问本地 DNS 缓存
    ↓ 没找到 ↓
    ↓ ③ 问本地 DNS 服务器（你家的路由器/ISP 的 DNS）
    ↓ 没找到 ↓
    ↓ ④ 本地 DNS 服务器去问根 DNS：" .com 的 DNS 在哪？"
    ↓ ⑤ 根 DNS："去问 .com 的顶级域名服务器"
    ↓ ⑥ .com DNS："baidu.com 的权威 DNS 是 ns1.baidu.com"
    ↓ ⑦ baidu.com 的权威 DNS："www.baidu.com 的 IP 是 110.242.68.66"
    ↓
最终返回: 110.242.68.66
```

**整个过程的耗时通常只有几十毫秒。** 因为层层缓存——你的电脑、路由器、ISP 都把上次查到的结果记下来了。

**网安意义：**
- DNS 劫持：把域名解析到假的 IP → 你访问的"百度"其实是钓鱼站
- DNS 隧道：把数据藏在 DNS 请求里传出去 → 绕过防火墙（因为防火墙通常允许 DNS）
- DNS Rebinding：先返回正常 IP 绕过同源策略，再改成内网 IP 攻击内网

### 🔍 ARP——局域网里的"谁是这个 IP？"

```
你的电脑（192.168.1.5）要发数据给路由器（192.168.1.1）

你的电脑广播 ARP 请求:
  "FF:FF:FF:FF:FF:FF（广播地址，发给所有人！）
   谁是 192.168.1.1？告诉我你的 MAC 地址。"

路由器回复:
  "是我！我的 MAC 地址是 aa:bb:cc:dd:ee:ff"

你的电脑记下这个映射（ARP 缓存），然后发帧过去。
```

**网安意义：ARP 欺骗。** 攻击者在局域网里不断发"我是网关！我的 MAC 是 xx:xx:xx"——其他机器被误导，把数据发给攻击者的 MAC 地址。攻击者成了"中间人"，可以看、改、丢弃所有流量。

## 7. 🧪 Wireshark 实战——30 分钟抓包实验

> **这是连接理论和现实的桥梁。必须做。**

### 安装 Wireshark

```bash
# Linux
sudo apt install wireshark

# Mac
brew install wireshark

# Windows
# 去 wireshark.org 下载安装包
# 安装时勾选 "Install Npcap"
```

### 实验 1：抓一次 HTTP 请求的全貌（15 分钟）

```bash
# 1. 打开 Wireshark，双击你的网卡开始抓包
# 2. 在浏览器里打开 http://httpbin.org/get（★注意是 http 不是 https）
# 3. Wireshark 里应该出现了大量包
# 4. 在 Filter 栏输入: http.host == httpbin.org
# 5. 你应该能看到 GET /get HTTP/1.1 的请求
# 6. 右键这个包 → Follow → TCP Stream
# 7. 你会看到完整的一来一回：
```

在 TCP Stream 窗口里你会看到类似：

```
GET /get HTTP/1.1
Host: httpbin.org
Connection: keep-alive
User-Agent: Mozilla/5.0 ...
Accept: text/html,application/xhtml+xml, ...

HTTP/1.1 200 OK
Date: Mon, 03 Jun 2026 10:30:00 GMT
Content-Type: application/json
Content-Length: 156

{
  "args": {},
  "headers": {
    "Host": "httpbin.org",
    ...
  },
  "origin": "你的IP地址",
  "url": "http://httpbin.org/get"
}
```

**你看到了什么？** 上面是请求（你发出去的），下面是响应（服务器回给你的）。中间一行空行分隔了 Headers 和 Body。

### 实验 2：看 TCP 三次握手（10 分钟）

```
1. 在 Wireshark 的 Filter 栏输入: tcp.port == 80 and http.host == httpbin.org
2. 清掉之前的抓包（Ctrl+Shift+Delete）
3. 刷新 http://httpbin.org/get 页面
4. 看前三个包:
   [SYN]      → 你的电脑 → 服务器     "在吗？"（SYN flag=1）
   [SYN, ACK] → 服务器 → 你的电脑     "在的"（SYN=1, ACK=1）
   [ACK]      → 你的电脑 → 服务器     "好，开始说"（ACK=1）
5. 点击每个包，在 Details 面板里展开 TCP 部分
6. 看 seq 和 ack 号的变化
```

### 实验 3：对比 HTTP vs HTTPS（5 分钟）

```
1. 先抓 http://httpbin.org/get → Follow TCP Stream → 明文可见
2. 再抓 https://httpbin.org/get → Follow TCP Stream → 什么都看不到！（加密了）
```

**这就是 HTTPS 的意义——中间的任何人都看不到你的内容。**

## 8. 家庭网络拓扑——你的数据出门经过了什么

### 🏪 生活场景：你出门上班的路线

```
你的卧室（手机）→ 你家大门（WiFi路由器）→ 小区大门（ISP接入点）
→ 城市主干道（ISP骨干网）→ 百度大厦（百度服务器）
```

### 🔍 你的手机访问 baidu.com 的完整路线

```
┌──────────┐    WiFi     ┌──────────┐    光猫     ┌──────────┐
│  手机     │ ←────────→ │ 无线路由器 │ ←────────→ │  光猫     │
│ WiFi网卡  │   802.11   │ NAT/路由  │   网线      │ 信号转换  │
└──────────┘            └──────────┘            └──────────┘
                              │
                              │ PPPoE 拨号
                              ▼
                        ┌──────────┐
                        │ ISP 接入层 │  ← 电信/联通/移动的机房
                        │ 分配公网 IP │
                        └──────────┘
                              │
                              ▼
                        ┌──────────┐
                        │ 骨干路由器 │  ← 全国/全球路由
                        │ BGP/OSPF  │
                        └──────────┘
                              │
                              ▼
                        ┌──────────┐
                        │ 百度 IDC   │
                        │ 服务器集群  │
                        └──────────┘
```

### NAT（网络地址转换）——你家只有 1 个公网 IP

```
你家所有人（手机、电脑、平板、电视）共享同一个公网 IP

内网:                               外网:
192.168.1.10:54321 →                → 你家的公网 IP:12345 → baidu.com:80
192.168.1.11:54321 → 路由器做 NAT → 你家的公网 IP:12346 → baidu.com:80
192.168.1.12:54321 →                → 你家的公网 IP:12347 → baidu.com:80

路由器维护一张表:
内网 IP:Port  ↔  外网 Port
当 baidu 回复到 :12345 → 路由器查表 → 转发给 192.168.1.10:54321
```

**安全意义：** NAT 天然提供了"入站防火墙"的效果——外网不能主动连你的内网设备（除非你在路由器上设置了端口转发）。
