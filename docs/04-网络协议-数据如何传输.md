## 1. 📌 分层模型——为什么网络要分层？

### 🏢 生活类比：寄快递

```
你寄一个包裹的过程天然就是分层的：

  应用层  = 你写一封信，装进信封
  传输层  = 快递员在信封外套个快递袋，写上单号（保证不丢件）
  网络层  = 分拣中心看目的地城市，决定走哪条路线（寻址与路由）
  链路层  = 货车司机实际搬运包裹（物理传输）

每一层只管自己的事，不看里面的内容。
快递员不需要知道信里写了什么，分拣中心也不管快递袋里是什么。
```

### OSI 七层 vs TCP/IP 四层

```
┌─────────────────────────────────────────────────┐
│  OSI七层(教科书)      TCP/IP四层(实际)   数据单元     │
├─────────────────────────────────────────────────┤
│  7.应用层  ┐                              │
│  6.表示层  ├─ 你写的程序 ──→  应用层    ←  数据    │
│  5.会话层  ┘                              │
├────────────────────────────────────────────┤
│  4.传输层  ── 端口/可靠性 ──→  传输层    ←  段(Segment) │
│                          TCP/UDP              │
├────────────────────────────────────────────┤
│  3.网络层  ── IP地址/路由 ──→  网络层    ←  包(Packet) │
├────────────────────────────────────────────┤
│  2.链路层  ┐                              │
│  1.物理层  ┴─ MAC地址/电压 ──→ 网络接口层 ←  帧(Frame) │
└─────────────────────────────────────────────────┘
```

### 📦 数据封装过程（一层套一层）

```
你要发 "Hello" 给百度服务器：

应用层:    [Hello]
              ↓ 加 HTTP 头
传输层:   [TCP头 | Hello]                    ← 加了端口号(80)
              ↓ 加 IP 头
网络层:  [IP头 | TCP头 | Hello]               ← 加了IP地址(百度)
              ↓ 加 MAC 头
链路层: [MAC头 | IP头 | TCP头 | Hello | 校验]  ← 加了MAC地址(网关)
              ↓ 转成电信号
物理层: ~~~~~~ 010101010 ~~~~~~→  网线传输
```

**解封装就是反过来一层一层剥皮。**

## 2. 📌 TCP——可靠传输的秘密

### 🏢 生活类比：打电话确认

```
TCP = 打电话
  1. 拨号（建立连接）     → "喂？" / "嗯我在"
  2. 说话（传输数据）     → 你说一句，对方"嗯"一声确认听到
  3. 挂断（释放连接）     → "那我挂了" / "好拜拜"

UDP = 发短信
  直接发出去，对方收没收到你不知道。
```

### 🔥 三次握手

```
客户端                              服务器
   │                                   │
   │──── SYN (seq=x) ────────────────→│  ① "嘿，你在吗？"
   │                                   │
   │←── SYN+ACK (seq=y, ack=x+1) ───│  ② "在的，你呢？"
   │                                   │
   │──── ACK (ack=y+1) ─────────────→│  ③ "我也在，开始聊吧"
   │                                   │
   │<═══════ 数据传输开始 ════════>    │
```

**为什么是三次不是两次？** 防止已过期的连接请求突然传到服务器——两次握手的话，一个延迟到达的老请求会让服务器建立无效连接。

### 四次挥手

```
客户端                              服务器
   │                                   │
   │──── FIN ───────────────────────→│  ① "我说完了，准备收线"
   │                                   │
   │←── ACK ────────────────────────│  ② "知道了"(但可能还有数据没传完)
   │                                   │
   │←── FIN ────────────────────────│  ③ "我也说完了"
   │                                   │
   │──── ACK ───────────────────────→│  ④ "好的，拜拜"
   │                                   │
   │   (等2MSL后真正关闭)              │
```

**网安意义：** SYN Flood 攻击就是只发 SYN 不回应 ACK，让服务器维持大量半连接耗尽资源。

## 3. UDP与ICMP——不需要握手的协议

### UDP——简单粗暴

```
UDP 头只有 8 字节：
┌────────────┬────────────┬────────────┬────────────┐
│ 源端口(2B)  │ 目的端口(2B)│  长度(2B)  │ 校验和(2B) │
└────────────┴────────────┴────────────┴────────────┘
然后直接跟数据，没有连接建立，没有确认机制。
```

**用在哪？** DNS查询、视频直播、游戏——允许丢几个包，但不能卡。

### ICMP——网络的诊断工具

**ping 命令用的就是 ICMP。** 它发送 Echo Request，目标回应 Echo Reply。

**网安意义：** 信息收集时用 ping 判断目标是否在线；有些防火墙禁止 ICMP，可以用来探测防火墙策略。

## 4. 📌 HTTP——Web的通用语言

### 🏢 生活类比：餐厅点菜

```
HTTP 请求  = 你叫服务员："一份宫保鸡丁，微辣"
  → 方法（GET）：我要点菜
  → URL（/menu/chicken）：我要的是宫保鸡丁
  → Headers：微辣（特殊要求）
  → Body（POST时）：详细的定制说明

HTTP 响应  = 服务员端上来
  → 状态码 200：做好了
  → 状态码 404：厨房没有这道菜
  → 状态码 500：厨师摔锅了
```

### HTTP 请求格式

```
GET /index.html HTTP/1.1            ← 请求行: 方法 + 路径 + 版本
Host: www.example.com               ← 请求头: 告诉服务器我是来你这的
User-Agent: Mozilla/5.0             ← 我用什么浏览器
Accept: text/html                   ← 我接受什么格式
Cookie: session=abc123              ← 我的身份凭证
                                    ← 空行表示请求头结束
                                    ← (POST请求这里放请求体)
```

### HTTP 响应格式

```
HTTP/1.1 200 OK                     ← 状态行: 版本 + 状态码 + 描述
Server: nginx                       ← 服务器用的什么
Content-Type: text/html; charset=utf-8  ← 返回的内容类型
Content-Length: 1234                ← 内容长度
Set-Cookie: session=xyz789          ← 叫你存个新凭证
                                    ← 空行
<!DOCTYPE html>                     ← 响应体开始
<html>...</html>
```

### 常用状态码

| 状态码 | 含义 | 类比 |
|:--:|------|------|
| 200 | OK，正常 | 菜上来了 |
| 301/302 | 重定向 | "这道菜换到3号窗口取了" |
| 400 | 你的请求有问题 | "你点的什么我听不懂" |
| 401 | 没登录 | "请先买单再点菜" |
| 403 | 没权限 | "VIP才能进这个包房" |
| 404 | 找不到 | "没有这道菜" |
| 500 | 服务器炸了 | "厨师摔锅了" |

### HTTP 方法

| 方法 | 含义 | 类比 |
|------|------|------|
| GET | 获取数据 | 看菜单 |
| POST | 提交数据 | 下单 |
| PUT | 更新（全量） | 换整道菜 |
| DELETE | 删除 | 取消订单 |
| HEAD | 只拿响应头 | 问菜品有没有，不用端上来 |
| OPTIONS | 问支持什么方法 | 问服务员能做啥 |

## 5. HTTPS与TLS——加密传输

### 🏢 生活类比：密信通信

```
HTTP  = 明信片——途中任何人（路由器、WiFi管理员）都能看到内容
HTTPS = 带锁的盒子——途中只能看到盒子，到了对方才能用钥匙打开
```

### TLS 握手（简化版）

```
客户端                              服务器
   │                                   │
   │── ClientHello ────────────────→│  ① "我支持这些加密方式"
   │   (支持的加密套件列表)             │
   │                                   │
   │←── ServerHello + 证书 ───────│  ② "好，用这套"，然后亮出身份证
   │   (选一种加密方式 + 服务器证书)     │
   │                                   │
   │── 验证证书                         │
   │   生成会话密钥                     │
   │   ClientKeyExchange ──────────→│  ③ 用服务器公钥加密的会话密钥
   │                                   │
   │<═══ 此后所有通信用会话密钥加密 ═══>│
```

**网安意义：** 渗透测试中经常要分析 HTTPS 流量——需要用 BurpSuite 的代理证书做中间人解密。

## 6. 📌 DNS与ARP——网络世界的电话本

### DNS——域名→IP

```
你在浏览器输入: www.baidu.com
  ↓ 你的电脑不知道 baidu.com 在哪
  ↓ 问 DNS 服务器：www.baidu.com 的 IP 是什么？
  ↓ DNS 回复：110.242.68.66
  ↓ 然后你的电脑才去连 110.242.68.66
```

**网安意义：** DNS 劫持（把你引导到假的IP）、DNS 隧道（用DNS传数据绕过防火墙）。

### ARP——IP→MAC地址

```
在局域网里通信需要 MAC 地址：

你的电脑（192.168.1.5）想发给路由器（192.168.1.1）
  ↓ 但不知道路由器的 MAC 地址
  ↓ 广播 ARP 请求："谁是 192.168.1.1？告诉我你的 MAC 地址。"
  ↓ 路由器回复："是我！我的 MAC 是 aa:bb:cc:dd:ee:ff"
  ↓ 然后才能发数据帧过去
```

**网安意义：** ARP 欺骗——攻击者冒充网关，让全网的流量经过自己（中间人攻击）。

## 🔧 Wireshark 实战五分钟

### 安装

```bash
# Linux
sudo apt install wireshark

# Windows/Mac: 去 wireshark.org 下载
```

### 动手抓包

1. 打开 Wireshark，选择你的网卡
2. 浏览器访问 `http://httpbin.org/get`（用 HTTP 不要用 HTTPS，方便看明文）
3. 在 Wireshark 里找到那个包
4. 右键 → "Follow" → "TCP Stream" → 你会看到完整的 HTTP 请求和响应！
